nmap Network Scanning

What You’re Doing

Demonstrating network scanning fundamentals using nmap against authorized targets. You'll perform version detection and script scanning to identify services and potential vulnerabilities.

Why It Matters

Nmap is the industry-standard network scanner. Understanding its capabilities helps both attackers (who use it to map target networks) and defenders (who need to understand what attackers see). Version detection reveals exactly what software is running, which directly informs vulnerability research.

  1. Open a terminal in your Kali VM
  2. Run a version detection scan:
    nmap -sV scanme.nmap.org
    ▶ What this does

    Concept: The -sV flag tells Nmap to probe open ports and determine what service and version is running on each one. Without it, Nmap only tells you a port is open — with it, you learn what software is listening (e.g., Apache 2.4.7, OpenSSH 6.6.1p1).

    In your output: Look for the VERSION column in the results table. Each open port should show specific software and version numbers. This information is what makes vulnerability research possible.

  3. Note the open ports and service versions detected
  4. Run a default script scan:
    nmap -sC scanme.nmap.org
    ▶ What this does

    Concept: The -sC flag runs Nmap’s default set of scripts against discovered services. These scripts perform safe enumeration tasks like grabbing SSH host keys, checking HTTP headers, and detecting common misconfigurations.

    In your output: Compare this to the -sV output. You should see additional details like SSH host keys and HTTP page titles that weren’t in the version scan. Scripts extend Nmap from a port scanner into an information-gathering platform.

  5. Compare the output — what additional information did scripts reveal?
terminal
Key Flags
  • -sV — Version detection
  • -sC — Default scripts
  • -A — Aggressive (combines -sV, -sC, OS detection, traceroute)

netcat Banner Grabbing

What You’re Doing

Using netcat to manually connect to network services and capture their identification banners — the text services display when you connect.

Why It Matters

Service banners reveal software versions, hostnames, and configuration details. This low-level technique helps verify nmap results and often reveals information automated scanners miss. Netcat is the “Swiss Army knife” of networking.

  1. Connect to SSH port on scanme.nmap.org:
    nc -v scanme.nmap.org 22
    ▶ What this does

    Concept: nc (netcat) opens a raw TCP connection to a host and port. The -v flag enables verbose output so you can see the connection status. Port 22 is the standard SSH port.

    In your output: You should see a connection message followed by the SSH banner (something like SSH-2.0-OpenSSH_6.6.1p1). This banner reveals the exact software version — valuable intel for an attacker.

  2. Observe the banner that appears (SSH version string)
  3. Press Ctrl+C to disconnect
  4. Try connecting to port 80:
    nc -v scanme.nmap.org 80
    ▶ What this does

    Concept: Port 80 is the standard HTTP port. Unlike SSH, web servers don’t send a banner automatically — you need to send an HTTP request first. This teaches you that different protocols behave differently during reconnaissance.

    In your output: You’ll see a connection confirmation, but no banner yet. You need to type the HTTP request in the next step to get a response.

  5. Type HEAD / HTTP/1.0 and press Enter twice to get HTTP headers
    ▶ What this does

    Concept: HEAD / HTTP/1.0 is a raw HTTP request asking the server for headers only (no page content). The response reveals the web server software, version, and sometimes OS details. This is manual banner grabbing for HTTP — exactly what automated tools do behind the scenes.

dirb Directory Discovery

What You’re Doing

Using dirb to brute-force discover hidden directories and files on web servers — finding content that isn’t linked from the main site.

Why It Matters

Web apps often have hidden admin panels, backup files, or development directories. Directory brute-forcing finds these resources that attackers commonly exploit.

  1. Run dirb against the test target:
    dirb http://testphp.vulnweb.com
    ▶ What this does

    Concept: dirb sends HTTP requests for thousands of common directory and file names (from a wordlist) and reports which ones exist. It’s a brute-force approach to content discovery — finding pages the site doesn’t link to publicly.

    In your output: Watch for lines showing CODE:200 (the page exists) versus CODE:403 (forbidden but exists) and CODE:301 (redirect). Each discovered path is a potential attack surface.

  2. Watch it test common directory names
  3. Note which return CODE:200 (found)
  4. Let it run or press Ctrl+C after several results

nikto Vulnerability Scanning

What You’re Doing

Using nikto to scan web servers for known vulnerabilities, misconfigurations, and dangerous files.

Why It Matters

Nikto checks for 6,700+ dangerous files and 1,250+ outdated server versions. It quickly identifies low-hanging fruit that attackers commonly exploit.

  1. Run nikto against the test target:
    nikto -h http://testphp.vulnweb.com
    ▶ What this does

    Concept: nikto is a web server vulnerability scanner that checks for thousands of known issues: outdated software, dangerous default files, misconfigurations, and insecure headers. The -h flag specifies the target host.

    In your output: Each line prefixed with + is a finding. Look for server version disclosures, missing security headers, and OSVDB reference numbers. These findings would go into a real vulnerability assessment report.

  2. Watch the output — nikto reports findings as it scans
  3. Note types of findings: server info, outdated software, dangerous files
  4. Let it run for a few minutes

enum4linux SMB Enumeration

What You’re Doing

Exploring enum4linux capabilities for SMB/Windows enumeration. Since we don’t have a live SMB target, you’ll review the tool’s help output and understand its purpose.

Why It Matters

SMB (Server Message Block) is ubiquitous in Windows networks. Enum4linux extracts usernames, shares, group memberships, and password policies — critical information for both attackers and security assessments. You’ll use this tool against the Brazos Financial container next week.

  1. Review enum4linux help:
    enum4linux -h
    ▶ What this does

    Concept: enum4linux is a tool for enumerating information from Windows/Samba systems using the SMB protocol. The -h flag displays all available options. We’re reviewing the help now because you’ll use this tool against a live target in Lab 1.3.

    In your output: Focus on the flags listed in the Key Flags callout below. Each flag extracts a different type of information from Windows systems — usernames, file shares, password policies, and group memberships.

  2. Note the different enumeration options (-U for users, -S for shares, etc.)
  3. Understand what each flag discovers
terminal
Key Flags
  • -U — Get user list
  • -S — Get share list
  • -P — Get password policy
  • -a — Do all simple enumeration

6
searchsploit Vulnerability Research

What You’re Doing

Using searchsploit to search the Exploit Database for known vulnerabilities and exploits based on software versions discovered during reconnaissance.

Why It Matters

Once you know what software a target runs, you need to research whether known exploits exist. Searchsploit provides offline access to Exploit-DB, the largest public archive of exploits and vulnerable software.

  1. Update the database:
    sudo searchsploit -u
    ▶ What this does

    Concept: The -u flag updates your local copy of the Exploit Database. Searchsploit works offline by searching a local mirror — updating ensures you have the latest exploits. New vulnerabilities are published daily, so keeping this current matters.

  2. Search for Apache vulnerabilities:
    searchsploit apache 2.4
    ▶ What this does

    Concept: This searches the local Exploit-DB for any exploit matching “apache 2.4.” In a real engagement, you’d take the exact version from your Nmap scan (e.g., Apache 2.4.7) and search for known exploits targeting it. This is where reconnaissance becomes actionable.

    In your output: You’ll see a table of matching exploits with titles, types (local, remote, webapps, DoS), and file paths. The path column tells you where the exploit code lives on your system if you need to examine it.

  3. Note the exploit titles, types, and paths
  4. Try searching for OpenSSH:
    searchsploit openssh
    ▶ What this does

    In your output: Compare the results to your Apache search. Notice the different exploit types and how the version numbers in the titles help you narrow down which exploits apply to a specific target. In practice, you’d cross-reference these results with the exact version string from your Nmap scan.
