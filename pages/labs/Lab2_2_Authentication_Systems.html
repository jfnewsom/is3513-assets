SSH Key Generation

What You’re Doing

Generating SSH keypairs using both RSA and ED25519 algorithms. You’ll compare the two key types and understand when to use each.

Why It Matters

SSH key authentication is the industry standard for secure remote access. Unlike passwords, keys are cryptographically strong, can’t be guessed through brute force, and provide non-repudiation. Every server you’ll manage in your career will use key-based authentication.

“ED25519 is the modern choice — smaller keys, faster operations, same security. But you’ll still see RSA everywhere. Know both.”

Derek Mitchell
Blue Team Lead, NEXUS Security

Part A: Generate an RSA keypair

  1. Create a working directory for this lab:
    mkdir -p ~/lab2-auth
  2. Change into your working directory:
    cd ~/lab2-auth
  3. Generate a 4096-bit RSA key (press Enter for defaults, use passphrase nexus2026):
    ssh-keygen -t rsa -b 4096 -f ./id_rsa_lab -C "lab-rsa-key"
    What this does

    Concept: ssh-keygen generates SSH keypairs for passwordless authentication. Like the RSA keys from Lab 2.1, SSH keys use asymmetric cryptography — but formatted for the SSH protocol. The private key stays on your machine; the public key goes on any server you want to access.

    In your output: You’ll see a randomart image — a visual fingerprint of your key. Two files are created: id_rsa_lab (private, keep secret) and id_rsa_lab.pub (public, safe to share).

    Flags:

    • -t rsa — Key type: RSA (the classic, widely compatible algorithm)
    • -b 4096 — Key size: 4096 bits (double the NIST minimum for stronger security)
    • -f ./id_rsa_lab — Output file path (avoids overwriting your real SSH keys)
    • -C "lab-rsa-key" — Comment embedded in the key — helps identify which key is which on servers with many authorized keys
  4. List the generated files:
    ls -la id_rsa*
  5. View the public key contents:
    cat id_rsa_lab.pub

Part B: Generate an ED25519 keypair

  1. Generate an ED25519 key (modern, faster, smaller):
    ssh-keygen -t ed25519 -f ./id_ed25519_lab -C "lab-ed25519-key"
    What this does

    Concept: ED25519 is a modern elliptic curve algorithm that provides equivalent security to RSA-4096 with much smaller keys and faster operations. Most security professionals now recommend ED25519 for new deployments — shorter keys, faster math, no known weaknesses. You’ll still encounter RSA on legacy systems, which is why you learned both.

  2. List all key files:
    ls -la id_*
  3. Compare the key sizes:
    wc -c id_rsa_lab.pub id_ed25519_lab.pub
    What this does

    In your output: wc -c counts bytes in each file. The RSA public key will be roughly 750 bytes while ED25519 is about 100 bytes — yet they provide comparable security. Smaller is better when security is equal.

Key Permissions

What You’re Doing

Understanding and configuring proper file permissions for SSH keys. You’ll see why SSH refuses to use keys with incorrect permissions and how to fix common permission problems.

Why It Matters

A private key with world-readable permissions is a security disaster. SSH clients will refuse to use improperly secured keys, and attackers specifically look for misconfigured key files. Proper permissions are the difference between security and breach.

Part A: Examine default permissions

  1. Check the current permissions on your keys:
    ls -la ~/lab2-auth/id_*
    What this does

    Concept: In Linux, every file has three permission groups: Owner, Group, and Others. Each group gets three bits: r (read=4), w (write=2), x (execute=1). Add them up per group to get octal notation. Private keys should be 600 (-rw-------), public keys 644 (-rw-r--r--), and the ~/.ssh directory 700 (drwx------).

    In your output: You should see -rw------- for private keys and -rw-r--r-- for public keys. The first column is the permission string.

  2. View the same permissions in octal notation:
    stat -c '%a %n' ~/lab2-auth/id_*
    What this does

    In your output: Private keys should show 600 and public keys 644 — confirming what ls -la showed. These are the correct, SSH-enforced permission levels.

Part B: Demonstrate the risk of weak permissions

  1. Copy the private key to a shared directory for this demonstration:
    cp ~/lab2-auth/id_ed25519_lab /tmp/demo_key
    What this does

    Concept: Your home directory (/home/kali) has permissions 700 — other users can’t enter it. /tmp is world-readable and world-writable, which is realistic: in enterprise environments, files often end up in shared directories like /tmp or /var/www. In those locations, file permissions are your only line of defense.

  2. Make the copy world-readable (INSECURE — for demonstration only):
    chmod 644 /tmp/demo_key
    What this does

    Concept: You’re changing this key from 600 (owner read/write only) to 644 (owner read/write, group can read, others can read). The permissions change from -rw------- to -rw-r--r--. That r-- at the end means every user on this machine can now read this file — fine for public keys, a disaster for private keys.

  3. Prove it — read the key as a completely different user:
    sudo -u nobody cat /tmp/demo_key
    What this does

    Concept: sudo -u nobody runs the following command as the nobody account — a minimal, unprivileged account with no special access to your files. Think of it as a stand-in for any random user or compromised process on this machine.

    In your output: You should see the entire private key printed to your terminal, starting with -----BEGIN OPENSSH PRIVATE KEY-----. This is the actual cryptographic secret. Anyone who has this data can impersonate you on any system where this key is authorized.

Screenshot

Part C: Fix the permissions and verify

  1. Restrict the file back to owner-only:
    chmod 600 /tmp/demo_key
    What this does

    Concept: chmod 600 sets permissions to -rw-------: 6 = read+write for owner, 0 = nothing for group, 0 = nothing for others. The change takes effect immediately — the kernel enforces it on the very next access attempt.

  2. Try reading the key as nobody again:
    sudo -u nobody cat /tmp/demo_key
    What this does

    In your output: You should now see: cat: /tmp/demo_key: Permission denied. Same command, same user, same file — only three digits changed (644 → 600). This is the principle of least privilege in action.

  3. Clean up the demo file:
    rm /tmp/demo_key
    What this does

    Concept: Good security hygiene: don’t leave copies of sensitive material in shared directories, even in a lab.

  4. Confirm your original keys still have correct permissions:
    stat -c '%a %n' ~/lab2-auth/id_*
    What this does

    In your output: Private keys should show 600 and public keys 644. The demonstration only affected the copy in /tmp, not your originals.

photo_camera
Screenshot

Capture the Permission denied output from the nobody user, along with the stat output confirming your original keys are still secure. Save as 13-permission-fixed.png

warning
Common Pitfalls
  • Leaving copies in /tmp: Always clean up after demonstrations
  • Wrong directory permissions: The ~/.ssh directory itself must be 700 (drwx------)
  • Public vs private: Public keys are designed to be shared (644). Private keys must always be 600
  • Defense in depth: Set both directory and file permissions correctly — don’t rely on a single layer

SSH Configuration

What You’re Doing

Setting up SSH infrastructure: the authorized_keys file that controls who can connect, the SSH config file that simplifies connections, and testing key-based authentication to localhost.

Why It Matters

Understanding SSH configuration is essential for any security professional. The authorized_keys file is your access control list — every public key there can authenticate. The config file lets you manage connections to dozens of servers efficiently. Gulf Coast uses this exact setup for their infrastructure.

“Never rely on a single control. Layer your defenses so one failure doesn’t mean total compromise. SSH config files let you enforce security policies across all your connections.”

Derek Mitchell
Blue Team Lead, NEXUS Security

Part A: Set up authorized_keys

  1. Create the .ssh directory if it doesn’t exist:
    mkdir -p ~/.ssh
  2. Set secure directory permissions:
    chmod 700 ~/.ssh
  3. Add your public key to authorized_keys:
    cat ~/lab2-auth/id_ed25519_lab.pub >> ~/.ssh/authorized_keys
    What this does

    Concept: The authorized_keys file is an access control list — every public key listed there can authenticate as this user. The >> operator appends rather than overwrites, preserving existing keys. This is how SSH key-based auth works: your client offers a private key, the server checks if the matching public key is in authorized_keys, and if so, grants access without a password.

  4. Set secure file permissions:
    chmod 600 ~/.ssh/authorized_keys
  5. Verify the directory contents:
    ls -la ~/.ssh/
  6. Confirm the authorized key:
    cat ~/.ssh/authorized_keys

Screenshot

Capture the .ssh directory listing and authorized_keys content. Save as 14-authorized-keys.png

Part B: Test key-based authentication

  1. Start the SSH service if not running:
    sudo systemctl start ssh
  2. Verify SSH is running:
    sudo systemctl status ssh
  3. Connect to localhost using your key:
    ssh -o PasswordAuthentication=no -i ~/lab2-auth/id_ed25519_lab localhost whoami
    What this does

    Concept: Adding a command after the hostname (whoami) tells SSH to run that single command on the remote system and return the output, instead of opening an interactive shell. This is how automated scripts and CI/CD pipelines use SSH.

    Flags:

    • -o PasswordAuthentication=no — Disables password fallback so the connection only succeeds with the key
    • -i — Specifies which private key to use (identity file)

Screenshot

Capture the successful SSH connection using key-based authentication. Save as 15-ssh-keyauth.png

Part C: Create an SSH config file

  1. Open a new config file in nano:
    nano ~/.ssh/config
    What this does

    Concept: nano is a simple terminal-based text editor. The commands at the bottom use ^ to mean the Ctrl key — so ^O means Ctrl+O. Since the file doesn’t exist yet, nano will create it when you save.

  2. Type or paste the following contents exactly (use spaces, not tabs for indentation):
    Host labserver HostName localhost User kali IdentityFile ~/lab2-auth/id_ed25519_lab IdentitiesOnly yes
    What this does

    Concept: Host labserver — the alias you’ll type. HostName localhost — the actual server address. User kali — the username. IdentityFile — which private key to use. IdentitiesOnly yes — only try the specified key, preventing SSH from cycling through all keys in ~/.ssh/.

  3. Save and exit nano: press Ctrl+O, then Enter to confirm the filename, then Ctrl+X to exit.
  4. Lock down config file permissions:
    chmod 600 ~/.ssh/config
    What this does

    Concept: Your SSH config contains key paths and server details. Setting 600 ensures only you can read or modify it — the same protection level as your private keys.

  5. Verify the config file contents:
    cat ~/.ssh/config
  6. Test the connection using your alias:
    ssh labserver whoami
    What this does

    In your output: Instead of typing the full command from Part B, you just typed ssh labserver whoami. The SSH config resolved labserver to all the connection details automatically. You should see your username with no password prompt.

  • Line breaks in key: The public key must be all on one line
  • Config indentation: Use spaces, not tabs; indent options under Host

Multi-Factor Authentication

What You’re Doing

Configuring your SSH server to require multi-factor authentication — both your private key and your account password before granting access. Right now, your server accepts key-only login. By the end of this checkpoint, a stolen key alone won’t be enough to get in.

Why It Matters

Single-factor authentication is the root cause of countless breaches. MFA isn’t optional anymore — it’s required by HIPAA, PCI-DSS, and most compliance frameworks. Authentication factors: something you know (passwords, PINs), something you have (private keys, hardware tokens), and something you are (fingerprints, facial recognition). True MFA requires factors from different categories.

“A password and a security question are both things you know — that’s two steps, not two factors. A private key and a password? That’s real MFA. One factor gets stolen, you still have a wall between the attacker and your system.”

Derek Mitchell
Blue Team Lead, NEXUS Security

Part A: Examine the current authentication policy

  1. Check what authentication methods your SSH server currently accepts:
    sudo sshd -T | grep -i authenticationmethods
    What this does

    Concept: sshd -T dumps the SSH server’s effective configuration (all settings, including defaults). Piping through grep -i authenticationmethods filters for the line that controls which authentication methods are required.

    In your output: If the output shows authenticationmethods any, the server accepts whatever works first — key or password. That’s single-factor.

Part B: Require both key and password

  1. Add the MFA directive to your SSH server configuration:
    echo "AuthenticationMethods publickey,password" | sudo tee -a /etc/ssh/sshd_config
    What this does

    Concept: This appends an MFA directive to the SSH server config. The syntax matters: a comma between methods means “AND” — the client must pass both. A space between methods means “OR” — the client can use either one. publickey,password = both required (MFA). publickey password = either one works (not MFA).

  2. Verify the line was added:
    tail -3 /etc/ssh/sshd_config
  3. Restart the SSH service to apply the change:
    sudo systemctl restart ssh

Part C: Test multi-factor login

  1. Connect using your SSH config alias:
    ssh labserver whoami
    What this does

    In your output: You should see two prompts in sequence: (1) Enter passphrase for key '...id_ed25519_lab': — unlocks your key file locally; (2) kali@localhost's password: — the server verifies your identity. Enter your key passphrase (nexus2026) at the first prompt, then your account password at the second. If both succeed, you’ll see your username from whoami.

Screenshot

Part D: Revert to single-factor

  1. Remove the MFA directive so later labs aren’t affected:
    sudo sed -i '/^AuthenticationMethods/d' /etc/ssh/sshd_config
  2. Restart SSH to apply changes:
    sudo systemctl restart ssh
  3. Verify single-factor is restored:
    sudo sshd -T | grep -i authenticationmethods
warning
Common Pitfalls
  • Comma vs. space: publickey,password = both required (MFA). publickey password = either one works (not MFA)
  • Forgot to restart SSH: Config changes don’t take effect until you restart the service
  • Key passphrase ≠ account password: Your passphrase (nexus2026) unlocks the key file locally. The account password is the separate factor the server verifies
