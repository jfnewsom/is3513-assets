What You're Doing

Generating SSH keypairs using both RSA and ED25519 algorithms. You'll compare the two key types and understand when to use each.

Why It Matters

SSH key authentication is the industry standard for secure remote access. Unlike passwords, keys are cryptographically strong, can't be guessed through brute force, and provide non-repudiation. Every server you'll manage in your career will use key-based authentication.

Derek Mitchell
Blue Team Lead, NEXUS Security

"ED25519 is the modern choice — smaller keys, faster operations, same security. But you'll still see RSA everywhere. Know both."

Step-by-Step Instructions

Part A: Generate an RSA keypair

  1. Create a working directory for this lab:
    mkdir -p ~/lab2-auth
  2. Change into your working directory:
    cd ~/lab2-auth
  3. Generate a 4096-bit RSA key (press Enter for defaults, use passphrase nexus2026):
    ssh-keygen -t rsa -b 4096 -f ./id_rsa_lab -C "lab-rsa-key"
    ▶ What this does

    Concept: ssh-keygen generates SSH keypairs for passwordless authentication. Like the RSA keys you made in Lab 2.1, SSH keys use asymmetric cryptography — but formatted specifically for the SSH protocol. The private key stays on your machine; the public key goes on any server you want to access.

    Flag breakdown:

    • -t rsa — Key type: RSA (the classic, widely compatible algorithm)
    • -b 4096 — Key size: 4096 bits (double the NIST minimum for stronger security)
    • -f ./id_rsa_lab — Output file path (avoids overwriting your real SSH keys)
    • -C "lab-rsa-key" — Comment embedded in the key — helps identify which key is which on servers with many authorized keys

    In your output: You’ll see a randomart image — a visual fingerprint of your key. Two files are created: id_rsa_lab (private, keep secret) and id_rsa_lab.pub (public, safe to share).

  4. List the generated files:
    ls -la id_rsa*
  5. View the public key contents:
    cat id_rsa_lab.pub

Part B: Generate an ED25519 keypair

  1. Generate an ED25519 key (modern, faster, smaller):
    ssh-keygen -t ed25519 -f ./id_ed25519_lab -C "lab-ed25519-key"
    ▶ What this does

    Concept: ED25519 is a modern elliptic curve algorithm that provides equivalent security to RSA-4096 with much smaller keys and faster operations. Most security professionals now recommend ED25519 over RSA for new deployments — shorter keys, faster math, no known weaknesses. You’ll still encounter RSA on legacy systems, which is why you learned both.

  2. List all key files:
    ls -la id_*
  3. Compare the key sizes:
    wc -c id_rsa_lab.pub id_ed25519_lab.pub
    ▶ What this does

    In your output: wc -c counts bytes in each file. The RSA public key will be roughly 750 bytes while ED25519 is about 100 bytes — yet they provide comparable security. This size difference matters at scale: when a server stores thousands of authorized keys, or when keys are exchanged over slow connections. Smaller is better when security is equal.

Common Pitfalls
  • Weak passphrase: Empty passphrases leave keys unprotected if stolen
  • Overwriting existing keys: Check for existing keys before generating new ones
  • Wrong key type: Some older systems don't support ED25519 — know your target

What You're Doing

Understanding and configuring proper file permissions for SSH keys. You'll see why SSH refuses to use keys with incorrect permissions and how to fix common permission problems.

Why It Matters

A private key with world-readable permissions is a security disaster. SSH clients will refuse to use improperly secured keys, and attackers specifically look for misconfigured key files. Proper permissions are the difference between security and breach.

Step-by-Step Instructions

Part A: Examine default permissions

  1. Check the current permissions on your keys:
    ls -la ~/lab2-auth/id_*

You should see output similar to this:

-rw------- 1 kali kal   464  Feb 5 19:05 id_ed25519_lab
-rw-r--r-- 1 kali kali   97  Feb 5 19:05 id_ed25519_lab.pub
-rw------- 1 kali kali 3414  Feb 5 18:57 id_rsa_lab
-rw-r--r-- 1 kali kali  737  Feb 5 18:57 id_rsa_lab.pub

Notice the first column — that’s the permission string. In Linux, every file has three permission groups:

Position Who Example
rwx------ Owner (you) Read, write, execute
---rwx--- Group (your user group) Read, write, execute
------rwx Others (everyone else) Read, write, execute

Each permission has a numeric value: r=4, w=2, x=1. Add them up per group to get the octal notation you’ll use with chmod:

Symbolic Octal Meaning Used For
-rw------- 600 Owner can read/write. Nobody else can touch it. Private keys
-rw-r--r-- 644 Owner can read/write. Everyone else can only read. Public keys
drwx------ 700 Owner has full access. Nobody else can enter. ~/.ssh directory
  1. Now view the same permissions in octal notation:
    stat -c '%a %n' ~/lab2-auth/id_*

This confirms what ls -la showed you — private keys are 600 (owner-only) and public keys are 644 (world-readable).

Part B: Demonstrate the risk of weak permissions

In Part A you learned that your private keys are set to 600 (-rw-------) — meaning only you can read or write them. Now you’re going to intentionally weaken those permissions and see exactly what happens when other users on the system can access your private key.

  1. Copy the private key to a shared directory for this demonstration:
    cp ~/lab2-auth/id_ed25519_lab /tmp/demo_key
    ▶ Why /tmp?

    Concept: Your home directory (/home/kali) has permissions 700 (drwx------). Remember from the permissions table in Part A: 700 means only the owner can read, write, or enter the directory. Other users can’t even list its contents, let alone read files inside it. This is defense in depth — even if a file inside has weak permissions, the directory permissions block access first.

    The /tmp directory, on the other hand, is world-readable and world-writable — every user and process on the system can access it. This is realistic: in enterprise environments, files often end up in shared directories like /tmp, /var/www, shared project folders, or cloud storage mounts. In those locations, file permissions are your only line of defense.

  2. Make the copy world-readable (INSECURE — for demonstration only):
    chmod 644 /tmp/demo_key
    ▶ What this does

    Concept: Remember the octal table from Part A? You’re changing this key from 600 (owner read/write, nobody else gets anything) to 644 (owner read/write, group can read, others can read). In the symbolic notation you saw with ls -la, the permissions change from -rw------- to -rw-r--r--. That r-- in the last three positions means every user on this machine can now read this file. This is the same permission level your public keys have — which is fine for public keys (they’re meant to be shared). But a private key with 644 permissions is like leaving your house key under the doormat and posting about it on social media.

  3. Now prove it — read the key as a completely different user:
    sudo -u nobody cat /tmp/demo_key
    ▶ What this does

    Concept: Let’s break this command down piece by piece:

    • sudo — Run this command with elevated privileges (you’ll enter your password)
    • -u nobody — “Run it as the user called nobody.” The nobody account exists on every Linux system — it’s a minimal, unprivileged account often used by background services. It has no special access to your files. Think of it as a stand-in for “any random user or compromised process on this machine.”
    • cat /tmp/demo_key — Print the contents of the file to the terminal

    In your output: You should see the entire private key dumped to your terminal, starting with -----BEGIN OPENSSH PRIVATE KEY----- followed by lines of Base64-encoded data. This is the actual cryptographic secret. Anyone who has this data can impersonate you on any system where this key is authorized.

    Real-world impact: On a shared server (which is common in enterprise environments), a compromised web server process, a rogue contractor’s account, or even a misconfigured backup script could silently copy your private key. They wouldn’t need your password. They wouldn’t need root access. They just need read access — which 644 gives to everyone.

Part C: Fix the permissions and verify

You just watched another user read your most sensitive credential. The entire private key — the one thing that proves you are you to every server that trusts it — was printed to the screen for anyone to copy. Now fix it and prove the fix actually works.

  1. Restrict the file back to owner-only:
    chmod 600 /tmp/demo_key
    ▶ What this does

    Concept: chmod 600 sets the permissions back to -rw-------. Breaking down the octal digit by digit: 6 = read (4) + write (2) for the owner, 0 = nothing for the group, 0 = nothing for others. You’re revoking the read access you granted in Part B and locking the file back down to owner-only. The change takes effect immediately — no restart, no reload. The kernel enforces it on the very next access attempt.

  2. Try reading the key as nobody again:
    sudo -u nobody cat /tmp/demo_key
    ▶ What this does

    In your output: This time you should see:

    cat: /tmp/demo_key: Permission denied

    What changed: Same command, same user, same file — the only difference is three digits (644 → 600). The Linux kernel checks permissions on every single file operation. The nobody user falls into the “others” permission group, and with 600, others get zero access — no read, no write, no execute. No application, script, or process running as another user can bypass this without root privileges. This is the principle of least privilege in action: give access only to who needs it, and nothing more.

  3. Clean up the demo file:
    rm /tmp/demo_key
    ▶ Why clean up?

    Concept: Good security hygiene means not leaving copies of sensitive material lying around, even in a lab. Your original key is still safe in ~/lab2-auth/ with proper 600 permissions, protected by both the file permissions and your home directory’s 700 permissions. That’s two layers of access control — defense in depth.

  4. Confirm your original keys still have correct permissions:
    stat -c '%a %n' ~/lab2-auth/id_*
    ▶ What to look for

    In your output: Private keys should show 600 and public keys should show 644 — exactly what you saw in Part A. The demonstration only affected the copy in /tmp, not your originals. This is another good habit: when testing something destructive, work on a copy so you can’t accidentally damage the real thing.

Screenshot #12

Capture the Permission denied output from the nobody user, along with the stat output confirming your original keys are still secure. Save as 12-permission-fixed.png

Common Pitfalls
  • Leaving copies in /tmp: Always clean up after demonstrations. Copies of private keys in shared directories are exactly the vulnerability you just demonstrated
  • Wrong directory permissions: The ~/.ssh directory itself must be 700 (drwx------). If the directory is readable, attackers can list what’s inside even if individual files are locked down
  • Public vs private: Public keys are designed to be shared, so 644 is correct for them. Private keys must always be 600. If you mix these up, you either lock people out (public key too restrictive) or expose secrets (private key too permissive)
  • Defense in depth: Notice that your home directory’s 700 permissions would have blocked the nobody user even if the key was 644. That’s why we used /tmp for this demo — but don’t rely on a single layer of protection. Set both directory and file permissions correctly

What You're Doing

Setting up SSH infrastructure: the authorized_keys file that controls who can connect, the SSH config file that simplifies connections, and testing key-based authentication to localhost.

Why It Matters

Understanding SSH configuration is essential for any security professional. The authorized_keys file is your access control list — every public key there can authenticate. The config file lets you manage connections to dozens of servers efficiently. Gulf Coast uses this exact setup for their infrastructure.

Derek Mitchell
Blue Team Lead, NEXUS Security

"Never rely on a single control. Layer your defenses so one failure doesn't mean total compromise. SSH config files let you enforce security policies across all your connections."

Step-by-Step Instructions

Part A: Set up authorized_keys

  1. Create the .ssh directory if it doesn't exist:
    mkdir -p ~/.ssh
  2. Set secure directory permissions:
    chmod 700 ~/.ssh
  3. Add your public key to authorized_keys:
    cat ~/lab2-auth/id_ed25519_lab.pub >> ~/.ssh/authorized_keys
  4. Set secure file permissions:
    chmod 600 ~/.ssh/authorized_keys
    ▶ What this does

    Concept: The authorized_keys file is an access control list — every public key listed there can authenticate as this user. The >> operator appends rather than overwrites, so existing keys are preserved. This is how SSH key-based authentication works: your client offers a private key, the server checks if the matching public key is in authorized_keys, and if so, grants access without a password.

  5. Verify the directory contents:
    ls -la ~/.ssh/
  6. Confirm the authorized key:
    cat ~/.ssh/authorized_keys
Screenshot #14

Capture the .ssh directory listing and authorized_keys content. Save as 14-authorized-keys.png

Part B: Test key-based authentication

  1. Start the SSH service if not running:
    sudo systemctl start ssh
  2. Verify SSH is running:
    sudo systemctl status ssh
  3. Connect to localhost using your key:
    ssh -o PasswordAuthentication=no -i ~/lab2-auth/id_ed25519_lab localhost whoami
    ▶ What this does

    Concept: Adding a command after the hostname (whoami) tells SSH to run that single command on the remote system and return the output, instead of opening an interactive shell. This is how automated scripts and CI/CD pipelines use SSH.

    Flag breakdown:

    • -o PasswordAuthentication=no — Disables password fallback so the connection only succeeds with the key
    • -i — Specifies which private key to use (identity file). Without this flag, SSH tries default key locations like ~/.ssh/id_ed25519

Part C: Create an SSH config file

Instead of typing long SSH commands every time you connect, you can create a config file that stores all the connection details under a simple alias. This is how professional security analysts manage connections to dozens of servers.

  1. Open a new config file in nano:
    nano ~/.ssh/config
    ▶ What this does

    Concept: nano is a simple terminal-based text editor. It opens directly in your terminal — no separate window, no mouse needed. The commands at the bottom of the screen (like ^O for “Write Out” and ^X for “Exit”) use the caret symbol ^ to mean the Ctrl key. So ^O means press Ctrl+O. Since the file doesn’t exist yet, nano will create it when you save.

  2. Type or paste the following contents exactly:
    Host labserver
        HostName localhost
        User kali
        IdentityFile ~/lab2-auth/id_ed25519_lab
        IdentitiesOnly yes
    ▶ What each line means
    • Host labserver — The alias you’ll type instead of a full command. You can name this anything you want
    • HostName localhost — The actual server address to connect to (in this lab, you’re connecting to your own machine)
    • User kali — The username to log in as
    • IdentityFile ~/lab2-auth/id_ed25519_lab — Which private key to use for this connection
    • IdentitiesOnly yes — Only try the specified key, don’t cycle through others. This prevents SSH from trying every key in ~/.ssh/ and potentially locking you out after too many failed attempts

    Indentation matters: The four spaces before each option tell SSH that those settings belong to the Host labserver block. Use spaces, not tabs.

  3. Save and exit nano:

    Press Ctrl+O (Write Out), then Enter to confirm the filename, then Ctrl+X to exit.

  4. Lock down config file permissions:
    chmod 600 ~/.ssh/config
    ▶ What this does

    Concept: Your SSH config file contains the paths to your private keys and your connection details. If another user could read it, they’d know exactly which keys to steal and which servers they grant access to. Setting 600 (-rw-------) ensures only you can read or modify this file — the same permission level as your private keys.

  5. Verify the config file contents:
    cat ~/.ssh/config
    ▶ What to look for

    In your output: You should see the five lines you typed in nano, exactly as shown above. If anything looks off (missing indentation, wrong key path, typos), run nano ~/.ssh/config again to fix it.

  6. Test the shortcut:
    ssh labserver whoami
    ▶ What this does

    In your output: Compare this to the full command you used in Part B. Instead of typing ssh -o PasswordAuthentication=no -i ~/lab2-auth/id_ed25519_lab localhost, you just typed ssh labserver whoami. The SSH config resolved labserver to all the connection details automatically. You should see your username printed with no password prompt. This is how professional environments work — consistent, repeatable, secure connections without memorizing complex command-line options.

  • Wrong key in authorized_keys: Make sure you're adding the .pub file, not the private key
  • Line breaks in key: The public key must be all on one line

What You’re Doing

Configuring your SSH server to require multi-factor authentication — both your private key and your account password before granting access. Right now, your server accepts key-only login. By the end of this checkpoint, a stolen key alone won’t be enough to get in.

Why It Matters

Single-factor authentication is the root cause of countless breaches. MFA isn’t optional anymore — it’s required by HIPAA, PCI-DSS, and most compliance frameworks. Authentication factors fall into three categories: something you know (passwords, PINs), something you have (private keys, hardware tokens), and something you are (fingerprints, facial recognition). True MFA requires factors from different categories. You’re about to implement one.

Derek Mitchell
Blue Team Lead, NEXUS Security

“A password and a security question are both things you know — that’s two steps, not two factors. A private key and a password? That’s real MFA. One factor gets stolen, you still have a wall between the attacker and your system.”

Step-by-Step Instructions

Part A: Examine the current authentication policy

  1. Check what authentication methods your SSH server currently accepts:
    sudo sshd -T | grep -i authenticationmethods
  2. If the output shows authenticationmethods any, the server accepts whatever works first — key or password. That’s single-factor.
▶ What this does

Concept: sshd -T dumps the SSH server’s effective configuration (all settings, including defaults). Piping through grep -i authenticationmethods filters for the line that controls which authentication methods are required. The -i flag makes the search case-insensitive.

Part B: Require both key and password

  1. Add the MFA directive to your SSH server configuration:
    echo "AuthenticationMethods publickey,password" | sudo tee -a /etc/ssh/sshd_config
    ▶ What this does

    Concept: This appends an MFA directive to the SSH server config. The syntax matters: a comma between methods means “AND” — the client must pass both in sequence. A space between methods means “OR” — the client can use either one.

    In practice: publickey,password requires both factors (MFA), while publickey password would accept either factor alone (not MFA).

  2. Verify the line was added:
    tail -3 /etc/ssh/sshd_config
  3. Restart the SSH service to apply the change:
    sudo systemctl restart ssh
Common Pitfalls
  • Comma vs. space: publickey,password = both required (MFA). publickey password = either one works (not MFA)
  • Forgot to restart SSH: Config changes don’t take effect until you restart the service
  • Key passphrase ≠ account password: Your key passphrase (nexus2026) unlocks the key file locally. The server never sees it. The account password is the separate factor the server verifies

Part C: Test multi-factor login

  1. Connect using your SSH config alias:
    ssh labserver whoami
  2. You should now see two prompts in sequence:
    Enter passphrase for key '/home/<user>/lab2-auth/id_ed25519_lab': ← unlocks your key file (local) <user>@localhost's password: ← server verifies your identity (remote)
  3. Enter your key passphrase at the first prompt, then your account password at the second. If both succeed, you’ll see your username printed by whoami.
Screenshot #17

Part D: Revert to single-factor

  1. Remove the MFA directive so later labs aren’t affected:
    sudo sed -i '/^AuthenticationMethods/d' /etc/ssh/sshd_config
  2. Restart SSH to apply changes:
    sudo systemctl restart ssh
  3. Verify single-factor is restored:
    sudo sshd -T | grep -i authenticationmethods
