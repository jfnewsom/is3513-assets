What You're Doing

Generating SSH keypairs using both RSA and ED25519 algorithms. You'll compare the two key types and understand when to use each.

Why It Matters

SSH key authentication is the industry standard for secure remote access. Unlike passwords, keys are cryptographically strong, can't be guessed through brute force, and provide non-repudiation. Every server you'll manage in your career will use key-based authentication.

Derek Mitchell
Blue Team Lead, NEXUS Security

"ED25519 is the modern choice — smaller keys, faster operations, same security. But you'll still see RSA everywhere. Know both."

Step-by-Step Instructions

Part A: Generate an RSA keypair

  1. Create a working directory for this lab:
    mkdir -p ~/lab2-auth cd ~/lab2-auth
  2. Generate a 4096-bit RSA key (press Enter for defaults, use passphrase nexus2026):
    ssh-keygen -t rsa -b 4096 -f ./id_rsa_lab -C "lab-rsa-key"
  3. View the generated files:
    ls -la id_rsa* cat id_rsa_lab.pub
Screenshot #9

Capture the RSA key generation showing the randomart and file listing. Save as 09-keygen-rsa.png

Part B: Generate an ED25519 keypair

  1. Generate an ED25519 key (modern, faster, smaller):
    ssh-keygen -t ed25519 -f ./id_ed25519_lab -C "lab-ed25519-key"
  2. Compare the two key types:
    ls -la id_* wc -c id_rsa_lab.pub id_ed25519_lab.pub
Screenshot #10

Capture the ED25519 generation and comparison showing both key types with their sizes. Notice the ED25519 public key is about 100 bytes while RSA is about 750 bytes. Save as 10-keygen-ed25519.png

Common Pitfalls
  • Weak passphrase: Empty passphrases leave keys unprotected if stolen
  • Overwriting existing keys: Check for existing keys before generating new ones
  • Wrong key type: Some older systems don't support ED25519 — know your target

What You're Doing

Understanding and configuring proper file permissions for SSH keys. You'll see why SSH refuses to use keys with incorrect permissions and how to fix common permission problems.

Why It Matters

A private key with world-readable permissions is a security disaster. SSH clients will refuse to use improperly secured keys, and attackers specifically look for misconfigured key files. Proper permissions are the difference between security and breach.

Step-by-Step Instructions

Part A: Examine default permissions

  1. Check the current permissions on your keys:
    ls -la ~/lab2-auth/id_*
  2. Note the difference between private key (600) and public key (644) permissions

Part B: Demonstrate permission failure

  1. Make the private key world-readable (INSECURE — for demonstration only):
    chmod 644 ~/lab2-auth/id_ed25519_lab ls -la ~/lab2-auth/id_ed25519_lab
  2. Attempt to use the key (SSH will refuse):
    ssh -i ~/lab2-auth/id_ed25519_lab localhost
Screenshot #11

Capture the "UNPROTECTED PRIVATE KEY FILE" error message from SSH. Save as 11-permission-error.png

Part C: Fix the permissions

  1. Restore correct permissions:
    chmod 600 ~/lab2-auth/id_ed25519_lab ls -la ~/lab2-auth/id_ed25519_lab
Screenshot #12

Capture the corrected permissions showing 600 on the private key. Save as 12-permission-fixed.png

Common Pitfalls
  • Forgetting to fix permissions: After demonstrating the error, always restore 600
  • Wrong directory permissions: ~/.ssh directory itself must be 700
  • Public vs private: Public keys can be 644, private keys MUST be 600
What You Should Have

Two screenshots: one showing the permission error, one showing the corrected permissions. This demonstrates why SSH is strict about key file security.

Textbook Reference

Chapter 11 covers access control and file permissions. The principle of least privilege applies directly to SSH keys — they should only be readable by their owner.

What You're Doing

Setting up SSH infrastructure: the authorized_keys file that controls who can connect, the SSH config file that simplifies connections, and testing key-based authentication to localhost.

Why It Matters

Understanding SSH configuration is essential for any security professional. The authorized_keys file is your access control list — every public key there can authenticate. The config file lets you manage connections to dozens of servers efficiently. Gulf Coast uses this exact setup for their infrastructure.

Derek Mitchell
Blue Team Lead, NEXUS Security

"Never rely on a single control. Layer your defenses so one failure doesn't mean total compromise. SSH config files let you enforce security policies across all your connections."

Step-by-Step Instructions

Part A: Set up authorized_keys

  1. Create the .ssh directory if it doesn't exist:
    mkdir -p ~/.ssh chmod 700 ~/.ssh
  2. Add your public key to authorized_keys:
    cat ~/lab2-auth/id_ed25519_lab.pub >> ~/.ssh/authorized_keys chmod 600 ~/.ssh/authorized_keys
  3. Verify the setup:
    ls -la ~/.ssh/ cat ~/.ssh/authorized_keys
Screenshot #13

Capture the .ssh directory listing and authorized_keys content. Save as 13-authorized-keys.png

Part B: Test key-based authentication

  1. Start the SSH service if not running:
    sudo systemctl start ssh sudo systemctl status ssh
  2. Connect to localhost using your key:
    ssh -i ~/lab2-auth/id_ed25519_lab localhost whoami
Screenshot #14

Capture the successful SSH connection using key-based authentication. Save as 14-ssh-keyauth.png

Part C: Create an SSH config file

  1. Create the config file:
    cat > ~/.ssh/config << 'EOF' Host labserver HostName localhost User $USER IdentityFile ~/lab2-auth/id_ed25519_lab IdentitiesOnly yes EOF chmod 600 ~/.ssh/config
  2. Test the config file:
    cat ~/.ssh/config ssh labserver whoami
Screenshot #15

Capture the SSH config file contents and successful connection using the alias. Save as 15-ssh-config.png

Common Pitfalls
  • Wrong key in authorized_keys: Make sure you're adding the .pub file, not the private key
  • Line breaks in key: The public key must be all on one line
  • Config indentation: Use spaces, not tabs; indent options under Host

What You're Doing

Understanding multi-factor authentication (MFA) concepts and the three authentication factor categories. You'll document why single-factor authentication is insufficient and how MFA provides defense in depth.

Why It Matters

Single-factor authentication is the root cause of countless breaches. MFA isn't just "nice to have" anymore — it's required by most compliance frameworks including HIPAA. Understanding the factor categories helps you design and evaluate authentication systems.

Step-by-Step Instructions

Part A: Document the three authentication factors

  1. Create an MFA documentation file:
    cat > ~/lab2-auth/mfa-factors.txt << 'EOF' AUTHENTICATION FACTORS ====================== SOMETHING YOU KNOW (Knowledge Factor) - Passwords - PINs - Security questions - Passphrases SOMETHING YOU HAVE (Possession Factor) - SSH private keys - Smart cards - Hardware tokens (YubiKey) - Mobile authenticator apps (TOTP) SOMETHING YOU ARE (Inherence Factor) - Fingerprint - Facial recognition - Iris scan - Voice recognition MFA = Using 2+ factors from DIFFERENT categories Examples: - Password + SSH key = MFA (know + have) - Password + security question = NOT MFA (both knowledge) - Fingerprint + facial = NOT MFA (both inherence) EOF
  2. View the documentation:
    cat ~/lab2-auth/mfa-factors.txt
Screenshot #16

Capture your MFA factors documentation showing all three categories. Save as 16-mfa-factors.png

Common Pitfalls
  • Confusing 2-step with 2-factor: Two passwords is 2-step but NOT MFA
  • SMS as "something you have": Debatable — SIM swapping makes SMS weak
  • Ignoring usability: MFA that users bypass defeats the purpose
What You Should Have

One screenshot showing your MFA factors documentation. This demonstrates understanding of the three categories and what constitutes true MFA.

Textbook Reference

Chapter 11 covers authentication factors in detail. The distinction between "something you know/have/are" is fundamental to understanding defense in depth for identity verification.

Dig Deeper

Research passwordless authentication. How do FIDO2/WebAuthn standards work, and why are they considered more secure than traditional MFA?
