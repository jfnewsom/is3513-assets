Step-by-Step Instructions

Step 1: Prepare Your Capture Environment

  1. Open Wireshark on your host machine (not inside the Kali VM)
  2. Identify your active network interface (Wi-Fi or Ethernet)
  3. First, resolve the server’s IP address. Open a terminal on your host and run:
    nslookup gulfcoast-portal.southcentralus.azurecontainer.io
    ▶ What this does

    Concept: nslookup queries the Domain Name System (DNS) to translate a hostname into an IP address. This is the same process your browser performs behind the scenes every time you visit a website.

    In your output: Look for a line showing Address: followed by an IP (e.g., 52.248.39.180). This is the IP you’ll use in your Wireshark capture filter.

    Note the IP address in the response (e.g., 52.248.39.180).
  4. Start a capture with that IP in your capture filter:
    host 52.248.39.180
    Replace the IP above with whatever nslookup returned.
Why Use the IP Address?

Wireshark’s capture filters (BPF filters) resolve hostnames once when the capture starts and can silently fail. Using the IP address directly is more reliable and is standard practice in network analysis. You’ll still use the FQDN for SSH and browser connections — only the capture filter needs the IP.

Step 2: Connect to Azure VM

  1. Open terminal in your Kali VM
  2. SSH to the Gulf Coast portal:
    ssh student@gulfcoast-portal.southcentralus.azurecontainer.io
    ▶ What this does

    Concept: SSH (Secure Shell) creates an encrypted tunnel between your Kali VM and the remote server. All traffic within the tunnel is encrypted, which is why Wireshark will show SSH packets but cannot read their contents. This is the same protocol used to remotely administer servers in production environments.

    Flag breakdown:

    • student — The username to authenticate as on the remote server
    • @ — Separates the username from the hostname
    • gulfcoast-portal...azurecontainer.io — The FQDN (Fully Qualified Domain Name) of the Azure-hosted server

    In your output: You’ll be asked to accept the host key fingerprint (type yes), then prompted for a password. After successful login, you’ll see a welcome banner identifying the Gulf Coast Healthcare Partners portal server.

  3. Password: NEXUSanalyst2026! (case-sensitive)
  4. Accept the host key fingerprint when prompted

Step 3: Generate Analysis Traffic

A real network analysis requires multiple protocol types in your capture. Complete all four activities below while your Wireshark capture is running. Do not stop the capture until you have completed all four.

Activity A — Browse via HTTPS (generates TLS traffic):

  1. Open a browser in your Kali VM (Firefox) and navigate to:
    https://gulfcoast-portal.southcentralus.azurecontainer.io
  2. Accept the certificate warning when prompted — this is a finding you will document
  3. Interact with the page (click links, explore)
Screenshot #21

Capture the Firefox certificate warning before you accept it. This self-signed certificate is a security finding. Save as 21-certificate-warning.png

Activity B — Browse via HTTP (generates unencrypted traffic):

  1. In the same browser, navigate to the HTTP version (no “s”):
    http://gulfcoast-portal.southcentralus.azurecontainer.io
  2. Notice anything different about how the site responds? Document your observation.

Activity C — Reconnaissance from Kali (generates HTTP header traffic):

  1. In your Kali VM (a separate terminal from your SSH session), request the HTTP headers from the unencrypted site:
    curl -I http://gulfcoast-portal.southcentralus.azurecontainer.io
    ▶ What this does

    Concept: curl is a command-line tool for making HTTP requests. In security assessments, it’s often preferred over a browser because it shows the raw server response without any client-side rendering or caching. Examining response headers is one of the first steps in web application reconnaissance.

    Flag breakdown:

    • -I — HEAD request — retrieves only the response headers, not the page content. This is less noisy than a full GET request and reveals server configuration details.

    In your output: You’ll see headers like Server:, Content-Type:, and others. Pay close attention — what information is the server revealing about itself that an attacker could use?

  2. Now request the headers from the HTTPS version:
    curl -k -I https://gulfcoast-portal.southcentralus.azurecontainer.io
    ▶ What this does

    Concept: This is the same request but over HTTPS (encrypted). The extra flag is needed because the server uses a self-signed certificate that curl wouldn’t normally trust. Comparing HTTP and HTTPS responses side-by-side can reveal configuration differences — servers sometimes expose different headers depending on the protocol.

    Flag breakdown:

    • -k — Insecure mode — tells curl to skip certificate verification. Without this, curl would refuse to connect because the certificate is self-signed (not issued by a trusted Certificate Authority).
    • -I — HEAD request — same as before, headers only.

    In your output: Compare this response carefully to the HTTP version. Do you see any headers here that were not present in the HTTP response? Any differences are potential security findings worth documenting.

Screenshot #22

Capture your terminal showing both curl outputs (HTTP and HTTPS headers) side by side or in sequence. Note the differences between the two responses. Save as 22-curl-header-comparison.png

Activity D — Server Exploration via SSH (generates interactive SSH traffic):

  1. In your SSH session, list the web server’s document root to see what sites are hosted:
    ls /var/www/
    ▶ What this does

    Concept: /var/www/ is the default web root on Linux servers. Listing its contents shows what web applications and directories the server hosts. During a security assessment, understanding the file structure helps identify potentially exposed content.

    In your output: You should see one or more directories. Note the names — do any suggest older or legacy systems that might have weaker security controls?

  2. Check for any Gulf Coast application files:
    ls /opt/gulfcoast-*
    ▶ What this does

    Concept: /opt/ is where third-party and custom applications are typically installed on Linux. The wildcard (*) matches any file or directory starting with gulfcoast-. Enumerating application directories helps map what software is running on the system.

    In your output: Any files or directories here represent installed Gulf Coast applications. Explore their contents for configuration files, logs, or other artifacts relevant to your assessment.

  3. Examine the web server configuration to understand how the site is set up:
    cat /etc/nginx/sites-enabled/*
    ▶ What this does

    Concept: /etc/nginx/sites-enabled/ contains the active site configurations for the nginx web server. Reading these files reveals how the server is configured: what ports it listens on, what SSL/TLS settings it uses, what locations are served, and what security headers (or lack thereof) are applied.

    In your output: This is a goldmine for your assessment. Read through the configuration carefully — look for SSL protocol versions, enabled locations, header settings, and anything that looks like a misconfiguration. Many of the findings you’ll document in your report can be confirmed right here.

  4. A good analyst checks for common reconnaissance targets. Think about what files a web server might expose that could reveal its structure.

Common Pitfalls
  • VPN blocking SSH: If you can't connect, disable any VPN on your host machine first
  • Wrong interface: Wireshark runs on your HOST machine — all other activities (browsing, SSH, curl) happen inside your Kali VM
  • Firewall issues: Some campus networks block outbound SSH (port 22) — try from a mobile hotspot
  • Password typo: Password is case-sensitive: NEXUSanalyst2026!
