Azure Connection & Live Capture

What You’re Doing

Establish an SSH connection to the Azure-hosted Gulf Coast patient portal and capture the network traffic using Wireshark on your host machine.

Why It Matters

This is your first interaction with real cloud infrastructure. Unlike training PCAPs, this is live traffic to a production-like environment. The skills you practiced in Lab 3.2 now apply to actual client systems. Jennifer Walsh is counting on your analysis to determine if the anomalous traffic her team flagged represents a genuine threat.

Step 1: Prepare Your Capture Environment

  1. Open Wireshark on your host machine (not inside the Kali VM).
  2. Identify your active network interface (Wi-Fi or Ethernet).
  3. Resolve the server’s IP address from a terminal on your host:
    nslookup gulfcoast-portal.southcentralus.azurecontainer.io
    What this does

    Concept: nslookup queries the Domain Name System (DNS) to translate a hostname into an IP address — the same process your browser performs behind the scenes every time you visit a website.

    In your output: Look for a line showing Address: followed by an IP (e.g., 52.248.39.180). This is the IP you’ll use in your Wireshark capture filter.

  4. Start a Wireshark capture on your active interface with a capture filter using the IP from nslookup:
    host 52.248.39.180
    (replace with the IP nslookup returned)
What this does

Concept: Wireshark’s capture filters (BPF filters) resolve hostnames once at capture start and can silently fail. Using the IP directly is more reliable and is standard professional practice. You’ll still use the FQDN for SSH and browser connections — only the capture filter needs the IP.

Step 2: Connect to Azure VM

  1. Open a terminal in your Kali VM.
  2. SSH to the Gulf Coast portal:
    ssh student@gulfcoast-portal.southcentralus.azurecontainer.io
    What this does

    Concept: SSH (Secure Shell) creates an encrypted tunnel between your Kali VM and the remote server. All traffic within the tunnel is encrypted, which is why Wireshark will show SSH packets but cannot read their contents.

    In your output: You’ll be asked to accept the host key fingerprint (type yes), then prompted for a password. After successful login, you’ll see a welcome banner identifying the Gulf Coast Healthcare Partners portal server.

    Flags:

    • student — The username to authenticate as on the remote server
    • @ — Separates the username from the hostname
    • gulfcoast-portal...azurecontainer.io — The FQDN (Fully Qualified Domain Name) of the Azure-hosted server
  3. Password: NEXUSanalyst2026! (case-sensitive). Accept the host key fingerprint when prompted.

Step 3 — Activity C: Reconnaissance from Kali

  1. In your Kali VM (a separate terminal from your SSH session), request the HTTP headers:
    curl -I http://gulfcoast-portal.southcentralus.azurecontainer.io
    What this does

    Concept: curl is a command-line tool for making HTTP requests. In security assessments, it’s preferred over a browser because it shows the raw server response without client-side rendering or caching. Examining response headers is one of the first steps in web application reconnaissance.

    In your output: You’ll see headers like Server:, Content-Type:, and others. Pay close attention — what information is the server revealing about itself that an attacker could use?

    Flags:

    • -I — HEAD request — retrieves only the response headers, not the page content. Less noisy than a full GET request and reveals server configuration details.
  2. Now request the HTTPS headers for comparison:
    curl -k -I https://gulfcoast-portal.southcentralus.azurecontainer.io
    What this does

    Concept: The same request but over HTTPS (encrypted). Comparing HTTP and HTTPS responses side-by-side can reveal configuration differences — servers sometimes expose different headers depending on the protocol.

    In your output: Compare this response carefully to the HTTP version. Do you see any headers here that were not present in the HTTP response? Any differences are potential security findings.

    Flags:

    • -k — Insecure mode — tells curl to skip certificate verification. Required here because the server uses a self-signed certificate not trusted by default.
    • -I — HEAD request — headers only, same as before.

Step 3 — Activity D: Server Exploration via SSH

  1. In your SSH session, list the web server’s document root:
    ls /var/www/
    What this does

    Concept: /var/www/ is the default web root on Linux servers. Listing its contents shows what web applications and directories the server hosts.

    In your output: You should see one or more directories. Note the names — do any suggest older or legacy systems that might have weaker security controls?

  2. Check for Gulf Coast application files:
    ls /opt/gulfcoast-*
    What this does

    Concept: /opt/ is where third-party and custom applications are typically installed. The wildcard (*) matches any file or directory starting with gulfcoast-.

    In your output: Any files or directories here represent installed Gulf Coast applications. Explore their contents for configuration files, logs, or other artifacts.

  3. Examine the web server configuration:
    cat /etc/nginx/sites-enabled/*
    What this does

    Concept: /etc/nginx/sites-enabled/ contains active site configurations for the nginx web server. Reading these files reveals how the server is configured: ports, SSL/TLS settings, served locations, and security headers (or lack thereof).

    In your output: This is a goldmine for your assessment. Read carefully — look for SSL protocol versions, enabled locations, header settings, and misconfigurations. Many findings you’ll document in your report can be confirmed right here.

warning
Common Pitfalls
  • VPN blocking SSH: If you can’t connect, disable any VPN on your host machine first
  • Wrong interface: Wireshark runs on your HOST machine — all other activities (browsing, SSH, curl) happen inside your Kali VM
  • Firewall issues: Some campus networks block outbound SSH (port 22) — try from a mobile hotspot
  • Password typo: Password is case-sensitive: NEXUSanalyst2026!
