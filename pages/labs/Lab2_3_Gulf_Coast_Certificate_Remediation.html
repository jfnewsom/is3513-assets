What You're Doing

Generating a new CSR using the existing private key, signing it with the CA, and installing the new certificate.

Why It Matters

Certificate renewal is one of the most common PKI operations you'll perform. Understanding the CSR workflow — using the existing private key to generate a new request, then signing with the CA — ensures continuity without regenerating keys.

Step-by-Step

  1. Explore PKI directory: ls -la /opt/gulfcoast-pki/
  2. Generate new CSR (see example commands below)
Example Commands
cd /opt/gulfcoast-pki
openssl req -new -key private/portal.key -out certs/portal-new.csr \
  -subj "/C=US/ST=Texas/L=Houston/O=Gulf Coast Healthcare Partners/CN=portal.gulfcoast.local"

openssl x509 -req -in certs/portal-new.csr -CA certs/ca.crt -CAkey private/ca.key \
  -CAcreateserial -out certs/portal-new.crt -days 365 -sha256

sudo cp /opt/gulfcoast-pki/certs/portal-new.crt /etc/ssl/certs/portal.crt

Capture CSR generation (gulfcoast-03-csr-generation.png), signing (gulfcoast-04-cert-signing.png), and installation (gulfcoast-05-cert-installation.png)

Common Pitfalls
  • Wrong key for CSR: The CSR must use the EXISTING portal.key, not generate a new one
  • Subject mismatch: CN (Common Name) must exactly match the server hostname
  • Permission errors: Certificate files need proper ownership and permissions for Apache

Step-by-Step

  1. Restart Apache: sudo systemctl restart apache2
  2. Verify new dates: openssl s_client -connect localhost:443 2>/dev/null | openssl x509 -noout -dates
  3. Test in browser: Refresh https://localhost:443 — should no longer show expired warning
  4. Exit container: exit

Common Pitfalls
  • Forgot to restart: Apache won't pick up the new certificate until restarted
  • Browser cache: Clear cache or use incognito if old certificate still shows
  • Still showing warning: May be self-signed trust issue, not expiration — that's expected
