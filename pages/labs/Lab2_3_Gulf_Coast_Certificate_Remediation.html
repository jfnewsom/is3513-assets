Step-by-Step Instructions

Part A: SSH into the container and explore the PKI structure

  1. SSH into the Gulf Coast portal container (password: nexus2026):
    ssh -p 2222 student@localhost
  2. Navigate to the PKI directory and read the documentation:
    cd /opt/gulfcoast-pki
  3. Read the README left by the previous administrator:
    cat README.txt
  4. Explore the directory structure:
    ls -la CA/ certs/ csr/
    ▶ What this does

    Concept: You’re mapping the PKI infrastructure before making changes. The CA/ directory holds the root Certificate Authority (ca.crt and ca.key). The certs/ directory holds the server certificates including the expired portal.crt. The csr/ directory is your working space for new certificate signing requests.

    Expected output: Note the file permissions — the CA key (ca.key) is root-owned for security, while the certs directory is student-owned so you can write new files there.

Part B: Generate a new Certificate Signing Request (CSR)

  1. Generate a new CSR using the existing private key:
    openssl req -new -key certs/portal.key -out csr/portal-new.csr -subj "/C=US/ST=Texas/L=Houston/O=Gulf Coast Healthcare Partners/CN=portal.gulfcoast.local"
    ▶ What this does

    Concept: A Certificate Signing Request (CSR) is like a formal application for a new certificate. It contains your public key and identity information, signed with your private key to prove ownership. The CA uses this to issue a new certificate without ever needing your private key.

    Flags: req -new creates a new CSR, -key specifies the existing private key to use, -out is the output file, -subj provides the identity fields inline instead of prompting interactively.

Screenshot #20

Capture the CSR generation command and output. Save as gulfcoast-03-csr-generation.png

Part C: Sign the CSR with the Certificate Authority

  1. Sign the CSR using the CA certificate and key (requires sudo because the CA key is root-owned):
    sudo openssl x509 -req -in csr/portal-new.csr -CA CA/ca.crt -CAkey CA/ca.key -CAcreateserial -out certs/portal-new.crt -days 365 -sha256
    ▶ What this does

    Concept: The CA “signs” your CSR by combining your identity and public key with the CA’s digital signature. Anyone who trusts the CA can now verify your certificate is legitimate. This is the chain of trust in action.

    Flags: x509 -req processes a CSR into a certificate, -CA and -CAkey specify the signing authority, -CAcreateserial generates a serial number, -days 365 sets one-year validity, -sha256 uses modern hashing.

    Expected output: You’ll see “Certificate request self-signature ok” followed by the subject line. The sudo password is nexus2026.

Screenshot #21

Capture the CA signing command and “self-signature ok” output. Save as gulfcoast-04-cert-signing.png

Part D: Install the new certificate

  1. Copy the new certificate into Apache’s SSL directory:
    sudo cp /opt/gulfcoast-pki/certs/portal-new.crt /etc/ssl/certs/portal.crt

Capture the certificate installation command. Save as gulfcoast-05-cert-installation.png

Common Pitfalls
  • Wrong key path: The portal key is at certs/portal.key — check the README and ls -la output to confirm paths
  • Permission denied on CA key: The CA private key is root-owned for security. You need sudo for the signing step
  • Subject mismatch: The CN (Common Name) must match the server hostname in the certificate
