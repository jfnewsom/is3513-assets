Deploy Container and Diagnose

What You’re Doing

Deploying the Gulf Coast patient portal container and documenting the broken state before making any changes.

Why It Matters

HIPAA requires documentation of security incidents and remediation. “Before” evidence proves due diligence for auditors. If you fix it first and document after, you have no proof of what the original problem was.

Part A: Deploy the Gulf Coast portal container

  1. Create a working directory for this engagement:
    mkdir -p ~/lab2-gulfcoast
  2. Change into your working directory:
    cd ~/lab2-gulfcoast
  3. Pull and start the Gulf Coast patient portal container:
    sudo docker run -d -p 443:443 -p 2222:22 --name gulfcoast-portal jfnewsom/gulfcoast-portal
    ▶ What this does

    Concept: This downloads and starts a Docker container simulating Gulf Coast’s patient portal — an Apache web server with a deliberately expired SSL certificate.

    Flags:

    • -d — Run in background (detached mode)
    • -p 443:443 — Map HTTPS port from container to host
    • -p 2222:22 — Map SSH to port 2222 so it doesn’t conflict with your Kali SSH service
    • --name gulfcoast-portal — Assign a friendly name for later docker commands
    • jfnewsom/gulfcoast-portal — The Docker Hub image to pull

Part B: Document the broken state

  1. Open Firefox and navigate to https://localhost — you should see a certificate warning. This is your “before” evidence.

  1. Check the certificate dates from your Kali terminal:
    openssl s_client -connect localhost:443 2>/dev/null | openssl x509 -noout -dates
    ▶ What this does

    Concept: This connects to the HTTPS server like a browser would, grabs the certificate, and displays its validity dates. The notBefore and notAfter dates tell you when the certificate was issued and when it expired.

    • -noout — Prevents printing the raw certificate
    • -dates — Shows only the validity period

Certificate Remediation

What You’re Doing

Generating a new CSR using the existing private key, signing it with the CA, and installing the new certificate.

Why It Matters

Certificate renewal is one of the most common PKI operations you’ll perform. Understanding the CSR workflow — using the existing private key to generate a new request, then signing with the CA — ensures continuity without regenerating keys.

Maria Delgado, PhD
Security Compliance Specialist, NEXUS Security

“The README in the PKI directory is your roadmap. Real infrastructure always has documentation — sometimes outdated, sometimes incomplete — but always your starting point. Read it before you touch anything.”

Part A: SSH into the container and explore the PKI structure

  1. SSH into the Gulf Coast portal container (password: nexus2026):
    ssh -p 2222 student@localhost
  2. Navigate to the PKI directory and read the documentation:
    cd /opt/gulfcoast-pki
  3. Read the README left by the previous administrator:
    cat README.txt
  4. Explore the directory structure:
    ls -la CA/ certs/ csr/
    ▶ What this does

    Concept: You’re mapping the PKI infrastructure before making changes. The CA/ directory holds the root Certificate Authority (ca.crt and ca.key). The certs/ directory holds the server certificates including the expired portal.crt. The csr/ directory is your working space for new certificate signing requests.

    In your output: Note the file permissions — the CA key (ca.key) is root-owned for security, while the certs/ directory is student-owned so you can write new files there.

Part B: Generate a new Certificate Signing Request (CSR)

  1. Generate a new CSR using the existing private key:
    openssl req -new -key certs/portal.key -out csr/portal-new.csr -subj "/C=US/ST=Texas/L=Houston/O=Gulf Coast Healthcare Partners/CN=portal.gulfcoast.local"
    ▶ What this does

    Concept: A Certificate Signing Request (CSR) is a formal application for a new certificate. It contains your public key and identity information, signed with your private key to prove ownership. The CA uses this to issue a new certificate without ever needing your private key.

    Flags:

    • req -new — Create a new CSR
    • -key certs/portal.key — Use the existing private key (no need to regenerate)
    • -out csr/portal-new.csr — Output file for the CSR
    • -subj — Provides identity fields inline (C=country, ST=state, L=locality, O=organization, CN=common name)
photo_camera
Screenshot

Capture the CSR generation command and output. Save as gulfcoast-03-csr-generation.png

Part C: Sign the CSR with the Certificate Authority

  1. Sign the CSR using the CA certificate and key (requires sudo because the CA key is root-owned):
    sudo openssl x509 -req -in csr/portal-new.csr -CA CA/ca.crt -CAkey CA/ca.key -CAcreateserial -out certs/portal-new.crt -days 365 -sha256
    ▶ What this does

    Concept: The CA “signs” your CSR by combining your identity and public key with the CA’s digital signature. Anyone who trusts the CA can now verify your certificate is legitimate. This is the chain of trust in action.

    In your output: You’ll see “Certificate request self-signature ok” followed by the subject line. The sudo password is nexus2026.

    Flags:

    • x509 -req — Process a CSR into a signed certificate
    • -CA / -CAkey — Specify the signing Certificate Authority and its key
    • -CAcreateserial — Generate a unique serial number for this certificate
    • -days 365 — Set one-year validity period
    • -sha256 — Use modern hashing algorithm for the signature

Screenshot

Capture the CA signing command and “self-signature ok” output. Save as gulfcoast-04-cert-signing.png

Part D: Install the new certificate

  1. Copy the new certificate into Apache’s SSL directory:
    sudo cp /opt/gulfcoast-pki/certs/portal-new.crt /etc/ssl/certs/portal.crt

warning
Common Pitfalls
  • Wrong key path: The portal key is at certs/portal.key — confirm with ls -la
  • Permission denied on CA key: The CA private key is root-owned. You need sudo for the signing step
  • Subject mismatch: The CN (Common Name) must match the server hostname in the certificate

warning
Common Pitfalls
  • SSH session disconnects on Apache restart — this is expected behavior inside Docker. Restart the container from your Kali terminal: sudo docker start gulfcoast-portal
  • Certificate change not reflected — if openssl still shows the old expiration date, run docker restart gulfcoast-portal and wait 10 seconds before checking again.
  • Self-signed trust warning persists in browser — this is separate from expiration. Your openssl verification is the authoritative check. The browser warning about self-signed CAs is expected and does not mean your fix failed.
  • Missing 'before' screenshots — you cannot recreate the broken state after fixing the certificate. If you missed them, note this limitation in your report.
