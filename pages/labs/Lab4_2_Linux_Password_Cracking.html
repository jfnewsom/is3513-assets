Key Concepts

  • SHA-512 crypt: Linux password hashing algorithm with built-in salting and key stretching (5000+ iterations)
  • Salt: Random data added to passwords before hashing, preventing rainbow table attacks
  • Key Stretching: Deliberately slow hashing through multiple iterations to increase crack time
  • Shared Accounts: Single credentials used by multiple users — common in OT environments, impossible to audit

Step-by-Step Instructions

Download and organize the Alamo Industries hash file:

Download alamo-hashes.txt and move it to your ~/lab4 directory from Lab 4.1.

Examine the hash file contents:

cat ~/lab4/alamo-hashes.txt
▶ What this does

Concept: These are Linux shadow-format hashes. Look for the $6$ prefix — that tells you the algorithm is SHA-512 crypt.

In your output: Each line follows username:$6$SALT$HASH format. The salt sits between the second and third $ signs — this is what NTLM was missing.

Isolate the first entry for closer inspection:

head -1 ~/lab4/alamo-hashes.txt
▶ What this does

Flag breakdown:

  • head -1 — Displays only the first line of a file

Concept: Compare this to NTLM from Lab 4.1: NTLM was 32 hex characters with no structure. SHA-512 crypt has algorithm ID ($6$), a unique salt, and an 86-character hash — dramatically more complex.

What You're Doing

Executing a dictionary attack against Alamo Industries' SHA-512 hashes using John the Ripper, documenting the dramatically slower crack speed.

Why It Matters

The speed difference you're about to witness is the entire point of modern password hashing. SHA-512 with salting and rounds is designed to be computationally expensive — what took seconds against NTLM may take hours here. This is defensive design in action, and understanding it helps you explain to clients why algorithm choice matters for their security posture.

Step-by-Step Instructions

Run John the Ripper against the SHA-512 hashes:

Note the start time — you will need to document the total cracking duration. John auto-detects the sha512crypt format from the $6$ prefix.

john --wordlist=/usr/share/wordlists/rockyou.txt ~/lab4/alamo-hashes.txt
▶ What this does

Flag breakdown:

  • --wordlist=... — Dictionary attack mode — tries every password in rockyou.txt
  • ~/lab4/alamo-hashes.txt — John reads the full shadow-format entry and auto-detects sha512crypt

In your output: Watch the H/s (hashes per second) rate — compare this to NTLM speeds from Lab 4.1. The difference is dramatic and intentional.

Pro Tip: SHA-512 is orders of magnitude slower than NTLM. Let the attack run — weak passwords will crack first. Press any key to see progress.

Step-by-Step Instructions

Run Hashcat with SHA-512 crypt mode:

Note the start time and observe the cracking speed. Since VMs typically lack GPU access, include the -D 1 flag to force CPU mode.

hashcat -m 1800 -a 0 -D 1 ~/lab4/alamo-hashes.txt /usr/share/wordlists/rockyou.txt
▶ What this does

Flag breakdown:

  • -m 1800 — Hash mode 1800 = sha512crypt (NOT 1000, which is NTLM)
  • -a 0 — Attack mode 0 = dictionary (straight wordlist)
  • -D 1 — Device type 1 = CPU only (required in most VMs)
  • ~/lab4/alamo-hashes.txt — Hashcat reads shadow-format entries directly — no need to extract hashes

In your output: Compare the H/s rate to both John on SHA-512 and your NTLM results from Lab 4.1. Even with GPU, SHA-512 remains dramatically slower — that is the whole point of modern hashing.

What Red Team Learns
  • Credential Access: SHA-512 significantly slows attacks, but weak passwords still fall. The algorithm buys time, not invincibility.
  • Target Prioritization: Shared accounts like shift_supervisor are high-value targets — one password, multiple users.
  • OT Environments: Manufacturing environments often prioritize operational continuity over security.
What Blue Team Learns
  • Defensive Architecture: Hash algorithm choice has massive security impact — SHA-512 is roughly 100,000x slower to crack than NTLM.
  • Shared Account Risk: Cannot enforce individual accountability with shared credentials.
  • Legacy Debt: Old accounts accumulate risk over time. Regular password audits prevent "common knowledge" passwords.
The Purple Team Connection

Comparing Windows and Linux password cracking reveals how defensive architecture choices affect attacker timelines. The same weak password pattern cracks instantly on NTLM but takes noticeable time on SHA-512. Architecture matters — but so does password policy. Frank Morrison's argument that "the password has worked fine for ten years" misses the point. Security isn't about whether something has failed yet — it's about whether it's resistant to attack.
