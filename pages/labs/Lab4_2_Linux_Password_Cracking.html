Key Concepts

  • SHA-512 crypt: Linux password hashing algorithm with built-in salting and key stretching (5000+ iterations)
  • Salt: Random data added to passwords before hashing, preventing rainbow table attacks
  • Key Stretching: Deliberately slow hashing through multiple iterations to increase crack time
  • Shared Accounts: Single credentials used by multiple users — common in OT environments, impossible to audit

Step-by-Step Instructions

  1. Download alamo-hashes.txt from Canvas to your ~/lab4 directory
  2. Examine the hash file contents:
    cat ~/lab4/alamo-hashes.txt
    ▶ What this does

    Concept: cat displays the full contents of the Alamo Industries hash file. Compare what you see here to the Brazos Financial hashes from Lab 4.1 — the format is dramatically different.

    In your output: You should see entries in username:$6$SALT$HASH format. The $6$ prefix identifies SHA-512 crypt. Notice each entry includes a unique salt between the second and third $ characters — this is the key architectural difference from NTLM.

  1. Isolate a single entry to analyze the format structure:
    head -1 ~/lab4/alamo-hashes.txt
    ▶ What this does

    Concept: head -1 displays only the first line of a file. Isolating one entry makes it easier to break down the format structure without visual clutter.

    In your output: Identify three components: the algorithm identifier ($6$ = SHA-512), the per-user salt (random characters between the second and third $), and the hash itself. This embedded metadata means the system knows exactly how to verify each password — and each user’s hash is unique even if two users share the same password.

What You're Doing

Executing a dictionary attack against Alamo Industries' SHA-512 hashes using John the Ripper, documenting the dramatically slower crack speed.

Why It Matters

The speed difference you're about to witness is the entire point of modern password hashing. SHA-512 with salting and rounds is designed to be computationally expensive — what took seconds against NTLM may take hours here. This is defensive design in action, and understanding it helps you explain to clients why algorithm choice matters for their security posture.

Step-by-Step Instructions

  1. Run John the Ripper against the SHA-512 hashes:
    john --wordlist=/usr/share/wordlists/rockyou.txt ~/lab4/alamo-hashes.txt
    ▶ What this does

    Concept: Same tool, same wordlist as Lab 4.1 — but a completely different experience. John auto-detects the $6$ prefix as sha512crypt format. No --format flag needed because the hash format is self-identifying (unlike NTLM’s raw hex).

    In your output: The speed difference will be dramatic. Where NTLM showed millions or billions of H/s, expect only thousands of H/s here. This is SHA-512 working as designed — the algorithm is deliberately slow to make brute-force attacks computationally expensive. Record the H/s rate carefully for your Lab 4.1 vs 4.2 comparison.

  1. Check progress while John is running:

    ▶ What this does

Step-by-Step Instructions

  1. Run Hashcat against the SHA-512 hashes:
    hashcat -m 1800 -a 0 -D 1 ~/lab4/alamo-hashes.txt /usr/share/wordlists/rockyou.txt
    ▶ What this does

    Concept: -m 1800 sets Hashcat to sha512crypt mode (compare to -m 1000 for NTLM in Lab 4.1). -a 0 runs a straight dictionary attack. -D 1 forces CPU mode since VMs typically lack GPU passthrough. Unlike Lab 4.1, Hashcat accepts the full shadow-format hashes — no need to extract hashes with cut first.

    In your output: Watch the Speed.#1 line. Compare it to Lab 4.1’s NTLM speed — the difference will be orders of magnitude. GPU acceleration provides minimal benefit for SHA-512 because the algorithm is designed to be memory-hard, not just compute-hard.

  1. Display cracked results:
    hashcat -m 1800 ~/lab4/alamo-hashes.txt --show
    ▶ What this does

    Concept: The --show flag displays cracked hashes from Hashcat’s potfile. Compare the number of cracked accounts to John’s results — they should be similar since both used the same wordlist against the same hashes. Any differences indicate where one tool continued running longer or used different cracking strategies.

What Blue Team Learns
  • Defensive Architecture: Hash algorithm choice has massive security impact — SHA-512 is roughly 100,000x slower to crack than NTLM.
  • Shared Account Risk: Cannot enforce individual accountability with shared credentials.
  • Legacy Debt: Old accounts accumulate risk over time. Regular password audits prevent "common knowledge" passwords.
The Purple Team Connection

Comparing Windows and Linux password cracking reveals how defensive architecture choices affect attacker timelines. The same weak password pattern cracks instantly on NTLM but takes noticeable time on SHA-512. Architecture matters — but so does password policy. Frank Morrison's argument that "the password has worked fine for ten years" misses the point. Security isn't about whether something has failed yet — it's about whether it's resistant to attack.
