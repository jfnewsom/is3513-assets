What You're Doing

Generating cryptographic hashes using SHA-256 and demonstrating the avalanche effect — how small input changes produce completely different outputs.

Why It Matters

Hashes are the foundation of data integrity. Every time you verify a software download, check file integrity, or validate a digital signature, you're using hashes. In healthcare, hash verification proves that patient records haven't been tampered with — critical for audit trails and legal compliance.

Maria Delgado, PhD
Security Compliance Specialist, NEXUS Security

"Auditors don't care what you did. They care what you can prove you did. Hash verification is how you prove data integrity."

Step-by-Step Instructions

Part A: Create a test file and generate its hash

  1. Create a working directory:
    mkdir -p ~/lab2-crypto
  2. Change into your working directory:
    cd ~/lab2-crypto
  3. Create a sample patient record file:
    echo "Patient: John Smith, DOB: 1985-03-15, MRN: 12345" > patient.txt
  4. Generate the SHA-256 hash:
    sha256sum patient.txt
    ▶ What this does

    Concept: sha256sum computes a SHA-256 cryptographic hash — a fixed-length, 64-character hexadecimal fingerprint of a file. The same input always produces the same hash, but even a one-bit change produces a completely different output. This is how you prove a file has not been tampered with.

    In your output: You should see a 64-character hex string followed by the filename. This hash is unique to your file’s exact contents. Copy it somewhere — you’ll compare it in the next step.

Screenshot #1

Capture the file creation and SHA-256 hash output. Save as 01-hash-sha256.png

Part B: Demonstrate the avalanche effect

  1. Create a slightly modified file (change one character):
    echo "Patient: John Smith, DOB: 1985-03-16, MRN: 12345" > patient2.txt
  2. Compare the hashes:
    sha256sum patient.txt patient2.txt
    ▶ What this does

    In your output: Compare the two hashes carefully. Despite changing only one character, the hashes are completely different. This is the avalanche effect — a critical property of cryptographic hash functions. If small changes produced similar hashes, attackers could predict modifications. The avalanche effect makes that impossible.

Capture both hashes side-by-side showing completely different outputs despite minimal input change. Save as 02-hash-comparison.png

Common Pitfalls
  • Using MD5 or SHA-1: These are deprecated for security purposes — always use SHA-256 or stronger
  • Confusing hashing with encryption: Hashes are one-way; you cannot reverse them to get original data

What You're Doing

Encrypting and decrypting files using AES-256, the symmetric encryption standard required by HIPAA for protecting patient health information (PHI) at rest.

Why It Matters

Symmetric encryption is how organizations protect bulk data. Every encrypted database, every secure file transfer, every protected backup uses symmetric encryption. AES-256 specifically is the gold standard — it's approved for classified government information and required by most healthcare compliance frameworks.

Maria Delgado, PhD
Security Compliance Specialist, NEXUS Security

"HIPAA requires encryption of PHI at rest. AES-256 is the standard. Know it cold."

Step-by-Step Instructions

Part A: Encrypt a file with AES-256

  1. Encrypt your patient file (use password nexus2026 when prompted):
    openssl enc -aes-256-cbc -salt -pbkdf2 -in patient.txt -out patient.txt.enc
    ▶ What this does

    Concept: This encrypts a file using AES-256 in CBC mode — the same algorithm the U.S. government uses for classified information. Symmetric encryption means the same password both encrypts and decrypts. HIPAA requires AES-256 for protecting patient health information (PHI) at rest.

    Flag breakdown:

    • enc — Tells OpenSSL to perform encryption/decryption
    • -aes-256-cbc — Algorithm: AES with 256-bit key in Cipher Block Chaining mode
    • -salt — Adds random data so the same password + file produces different ciphertext each time
    • -pbkdf2 — Uses PBKDF2 key derivation — makes brute-force password attacks much slower
    • -in / -out — Input plaintext file and output encrypted file

    In your output: OpenSSL will prompt for a password twice (enter and verify). Use nexus2026 as instructed. No output means success — OpenSSL is silent on success.

  2. Verify the encrypted file exists:
    ls -la patient.txt*
  3. View the encrypted contents as a hex dump:
    xxd patient.txt.enc | head
    ▶ What this does

    Concept: xxd is a hex dump utility that shows raw file bytes in hexadecimal format. Piping to head limits output to the first 10 lines. This lets you visually confirm the file is encrypted — you should see random-looking hex values, not readable text.

    In your output: The hex dump should show no recognizable text — just seemingly random bytes. The first 16 bytes contain the Salted__ header followed by the salt value. If you can read any of the original patient data, encryption failed.

Screenshot #3

Capture the encryption command and the hex dump showing unreadable encrypted content. Save as 03-symmetric-encrypt.png

Part B: Decrypt the file

  1. Decrypt using the same password:
    openssl enc -aes-256-cbc -d -pbkdf2 -in patient.txt.enc -out patient_decrypted.txt
    ▶ What this does

    Flag breakdown:

    • -d — Decrypt mode — this single flag reverses the encryption operation. Everything else stays the same: same algorithm, same key derivation. This demonstrates that symmetric encryption is reversible with the correct password.
  2. View the decrypted file:
    cat patient_decrypted.txt
  3. Confirm it matches the original:
    diff patient.txt patient_decrypted.txt && echo 'Files match!' || echo 'Files differ!'
Screenshot #4

Capture the decryption and verification showing the original content restored. Save as 04-symmetric-decrypt.png

Common Pitfalls
  • Forgetting the password: Symmetric encryption requires the same password for decryption
  • Using weak algorithms: DES is obsolete; always use AES-256
  • Missing -pbkdf2 flag: Modern OpenSSL requires this for key derivation

What You're Doing

Generating an RSA keypair and using it to encrypt and decrypt data. Unlike symmetric encryption (one shared key), asymmetric uses two mathematically related keys — a public key anyone can use to encrypt, and a private key only you possess to decrypt.

Why It Matters

Asymmetric encryption solves the key distribution problem. How do you securely share a symmetric key with someone you've never met? You use their public key. This is how TLS/HTTPS works, how secure email works, and how certificates establish trust.

Maria Delgado, PhD
Security Compliance Specialist, NEXUS Security

"Encryption is not magic. It's math. And math done wrong is worse than no math at all — it gives false confidence."

Step-by-Step Instructions

Part A: Generate an RSA keypair

  1. Generate a 2048-bit RSA private key:
    openssl genrsa -out private.pem 2048
    ▶ What this does

    Concept: This generates an RSA private key — the mathematical foundation for asymmetric cryptography. Unlike symmetric encryption (one shared secret), RSA uses a keypair: a private key you guard with your life, and a public key you share freely. The security comes from the mathematical difficulty of factoring large prime numbers.

    Flag breakdown:

    • genrsa — Generate an RSA private key
    • -out private.pem — Save to PEM format (Base64-encoded, human-readable headers)
    • 2048 — Key size in bits — NIST minimum for RSA. 4096 is more future-proof but slower.
  2. Extract the public key:
    openssl rsa -in private.pem -pubout -out public.pem
    ▶ What this does

    Concept: The public key is mathematically derived from the private key — but the reverse is computationally infeasible. This one-way relationship is what makes asymmetric crypto work: you can freely distribute your public key without exposing your private key. Anyone with your public key can encrypt messages that only your private key can decrypt.

  3. List both key files:
    ls -la *.pem
  4. View the public key contents:
    cat public.pem
Screenshot #5

Capture the keypair generation, file listing, and public key contents. Save as 05-rsa-keypair.png

Part B: Encrypt with public key

  1. Create a small message to encrypt:
    echo "Secret: AES key is nexus2026" > secret.txt
  2. Encrypt with the public key:
    openssl pkeyutl -encrypt -pubin -inkey public.pem -in secret.txt -out secret.enc
    ▶ What this does

    Concept: Public key encryption — anyone with your public key can encrypt a message, but only the private key holder can decrypt it. This solves the key distribution problem: you never need to share a secret password over an insecure channel.

    Flag breakdown:

    • pkeyutl — Public key utility for encrypt/decrypt/sign/verify operations
    • -encrypt — Perform encryption (use -decrypt to reverse)
    • -pubin — Input key is a public key (not a private key)
    • -inkey — Path to the key file to use
Screenshot #6

Capture the encryption command and verify the encrypted file exists. Save as 06-rsa-encrypt.png

Part C: Decrypt with private key

  1. Decrypt using the private key:
    openssl pkeyutl -decrypt -inkey private.pem -in secret.enc -out secret_decrypted.txt
  2. Verify the decrypted contents:
    cat secret_decrypted.txt

Capture the decryption showing the original secret message restored. Save as 07-rsa-decrypt.png

Common Pitfalls
  • Confusing public/private: Encrypt with public, decrypt with private (for confidentiality)
  • Data size limits: RSA can only encrypt small amounts of data directly
  • Exposing private key: Never share or commit private keys

What You're Doing

Creating and verifying digital signatures using RSA. Digital signatures use the private key to sign (proving authenticity) and the public key to verify (confirming identity). This is the inverse of encryption.

Why It Matters

Digital signatures provide non-repudiation — proof that a specific person signed a specific document. In healthcare, this is legally binding. When a physician signs an order electronically, that signature must be verifiable and undeniable.

Maria Delgado, PhD
Security Compliance Specialist, NEXUS Security

"Non-repudiation matters in healthcare. When a provider signs an order, it's legally binding."

Step-by-Step Instructions

Part A: Sign a document

  1. Create a document to sign:
    echo "Medication Order: Amoxicillin 500mg TID x 10 days - Dr. Chen" > order.txt
  2. Sign with your private key:
    openssl dgst -sha256 -sign private.pem -out order.sig order.txt
    ▶ What this does

    Concept: Digital signing is encryption in reverse — you sign with your private key, and anyone can verify with your public key. This provides non-repudiation: proof that a specific person signed a specific document. In healthcare, this is how electronic prescriptions and medical orders are legally binding.

    Flag breakdown:

    • dgst — Message digest (hash) operations
    • -sha256 — Hash the document with SHA-256 before signing
    • -sign private.pem — Sign using your private key — only you can create this signature
    • -out order.sig — Save the binary signature to a separate file
  3. Verify the signature file was created:
    ls -la order.*
Screenshot #8

Capture the signing command and file listing showing both the order and signature files. Save as 08-signature-create.png

Part B: Verify the signature

  1. Verify using the public key:
    openssl dgst -sha256 -verify public.pem -signature order.sig order.txt
    ▶ What this does

    In your output: You should see Verified OK — this confirms three things simultaneously: (1) the document has not been modified since signing, (2) the signature was created by the holder of the matching private key, and (3) the signer cannot deny having signed it (non-repudiation). If even one byte of order.txt changed, verification would fail.

Screenshot #9 (FINAL)

Capture the verification showing "Verified OK" message. Save as 09-signature-verify.png — this is your last screenshot for Lab 2.1!

Common Pitfalls
  • Confusing sign/verify with encrypt/decrypt: Sign with private, verify with public
  • Modified document: Any change to the document will cause verification to fail
