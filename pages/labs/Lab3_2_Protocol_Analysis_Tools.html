Install Wireshark and Identify Interfaces

What You’re Doing

Installing Wireshark on your host operating system and identifying which network interface to use for captures.

Why It Matters

Wireshark is the industry-standard network protocol analyzer. SOC analysts, incident responders, and penetration testers all rely on it daily. Understanding interface selection is critical — capturing on the wrong interface is the #1 beginner mistake.

  1. Download Wireshark from wireshark.org/download.html
  2. Install using default options (Windows: allow Npcap installation; macOS: grant permissions when prompted).
  3. Launch Wireshark and observe the interface list on the welcome screen.
  4. Identify your active interface by looking for the one with traffic (the sparkline graph shows activity).
  5. Note your interface name — you’ll need this for all future captures.

Verify Active Interface (Optional CLI Method)

  1. Windows (PowerShell):
    Get-NetAdapter | Where-Object Status -eq "Up"
  2. macOS/Linux:
    ifconfig | grep -E "^[a-z]"

Common Pitfalls
  • No interfaces visible: On Windows, run Wireshark as Administrator. On macOS, ensure you granted capture permissions during install.
  • VirtualBox interfaces showing: You’ll see “VirtualBox Host-Only” adapters — ignore these. Use your Wi-Fi or Ethernet interface.
  • Multiple active interfaces: If you have both Wi-Fi and Ethernet connected, pick whichever you use for internet access.

Learn Capture and Display Filters

What You’re Doing

Understanding the difference between capture filters and display filters, and practicing filter syntax.

Why It Matters

Large captures can contain millions of packets. Without filtering, finding relevant traffic is like searching for a needle in a haystack. Mastering filters is what separates efficient analysts from those who drown in data.

Filter Comparison

Aspect Capture Filter Display Filter
When applied During capture After capture
Syntax BPF (Berkeley Packet Filter) Wireshark-specific
Example: Port 80 port 80 tcp.port == 80
Example: Specific host host 10.0.0.1 ip.addr == 10.0.0.1

Practice Exercise

  1. Start a capture on your active interface (no filter).
  2. Let it run for 30 seconds while browsing a website.
  3. Stop the capture.
  4. Apply each of these display filters and observe the results:
    • dns — Show only DNS queries
    • http — Show only HTTP traffic
    • tcp.port == 443 — Show HTTPS traffic
    • ip.addr == YOUR_IP — Show only your traffic (replace with your IP)
lightbulb
Filter Syntax Helper

Display filters turn green when valid and red when invalid. If your filter bar is red, check your syntax. You can also use the Expression builder (bookmark icon next to the filter bar) to construct filters visually.

warning
Common Pitfalls
  • Mixing syntax: Don’t use display filter syntax in the capture filter box (or vice versa) — they’re different languages
  • Forgetting == vs =: Display filters use == for equality, not single =
  • Case sensitivity: Protocol names in display filters are lowercase (http not HTTP)

warning
Common Pitfalls
  • Mixing filter syntax: BPF capture filter syntax and Wireshark display filter syntax are different. Using display filter syntax in the capture filter box will fail silently.
  • No ICMP traffic: ICMP won't appear in normal browsing — you need to generate it. Run ping google.com in a terminal while Wireshark is capturing.
  • Missing protocol documentation: Don't just capture — document each protocol's purpose, whether it's encrypted, and its port numbers. That analysis becomes the core of your Lab 3.3 Tool Documentation section.
  • Case sensitivity in display filters: Protocol names in display filters are lowercase: use http, not HTTP. The filter bar turns green when valid, red when invalid.

Advanced Analysis Techniques

What You’re Doing

Learning advanced Wireshark features: Follow TCP Stream, Endpoint Statistics, IO Graphs, and Flow Graphs.

Why It Matters

Basic filtering shows individual packets. Advanced techniques reveal patterns, relationships, and behaviors. These are the tools that separate packet-by-packet analysis from true network investigation.

Technique 1: Follow TCP Stream

  1. Find an HTTP or TCP packet in your capture.
  2. Right-click → Follow → TCP Stream.
  3. Observe the full conversation (requests in red, responses in blue).
    What this does

    Concept: Reconstructs the complete conversation between two endpoints. Client requests appear in red, server responses in blue.

    In your output: This is how analysts read cleartext credentials in HTTP traffic.

Screenshot

Capture the TCP Stream window showing a reconstructed conversation. Note the color coding: client requests in red, server responses in blue. Save as 15-tcp-stream.png

Technique 2: Endpoint Statistics

  1. Go to Statistics → Endpoints.
  2. Click the IPv4 tab.
  3. Sort by “Packets” or “Bytes” to see top talkers.
    What this does

    In your output: Use case: Identify suspicious high-volume hosts during incident response.

photo_camera
Screenshot

Capture the Endpoints window with the IPv4 tab selected, sorted by packet count to show the top talkers. Save as 16-endpoints.png

Technique 3: IO Graphs

  1. Go to Statistics → I/O Graphs.
  2. Observe the default “All Packets” graph.
  3. Add a new line: Click +, enter filter dns, change color.
    What this does

    In your output: Use case: Spot traffic spikes, beaconing patterns, data exfiltration.

photo_camera
Screenshot

Capture the IO Graph showing at least two lines — the default “All Packets” and your custom DNS filter. Save as 17-io-graph.png

Technique 4: Flow Graphs

  1. Go to Statistics → Flow Graph.
  2. Select “Displayed packets” to use your current filter.
  3. Choose “TCP Flows” for flow type.
    What this does

    In your output: Use case: Understand handshakes, retransmissions, connection sequences.

warning
Common Pitfalls
  • Following encrypted streams: TCP stream on HTTPS shows gibberish — that’s encryption working. Try HTTP instead for readable content.
  • Empty IO Graphs: Make sure you have enough captured data. A 10-second capture won’t show meaningful patterns.
  • Overwhelming flow graphs: Apply a filter first to reduce the number of flows shown.
