Deploy Brazos Container

What You’re Doing

Deploying the Brazos Financial Docker container that simulates their external attack surface. This is your authorized target for the engagement.

Why It Matters

Real penetration tests begin with scope definition and authorized target deployment. The Brazos container represents their actual exposed services — the same ports and configurations an external attacker would see.

  1. Pull the Brazos container image:
    docker pull jfnewsom/brazos-financial:latest
    What this does

    Concept: docker pull downloads a container image from Docker Hub. This image contains Brazos Financial’s simulated environment — a pre-configured system with intentional vulnerabilities for you to discover. The :latest tag ensures you get the current version.

  2. Run the container with all required ports:
    docker run -d -p 80:80 -p 443:443 -p 22:22 -p 445:445 -p 8443:8443 --name brazos jfnewsom/brazos-financial:latest
    What this does

    Concept: docker run creates and starts a container. The -d flag runs it in the background (detached). Each -p flag maps a container port to your host: 80 (HTTP), 443 (HTTPS), 22 (SSH), 445 (SMB), and 8443 (alternate HTTPS). These are the services you’ll scan — Brazos Financial’s exposed attack surface.

  3. Verify the container is running:
    docker ps
    What this does

    In your output: You should see a table with the brazos container listed as “Up.” Verify all five port mappings appear in the PORTS column. If the STATUS shows “Exited,” the container crashed — check the Common Pitfalls below.

  4. Note the container IP (usually localhost or 127.0.0.1)

Full Reconnaissance Workflow

What You’re Doing

Conducting comprehensive reconnaissance against the Brazos Financial container using all six tools from your training: nmap, netcat, dirb, nikto, enum4linux, and searchsploit.

Why It Matters

This is the core of your engagement. A thorough reconnaissance reveals everything an attacker would discover about Brazos Financial’s external attack surface. Your findings drive the entire client report.

  1. Run comprehensive nmap scan:
    nmap -sV -sC -A localhost
    What this does

    Concept: This combines all three Nmap techniques from Lab 1.2: version detection (-sV), default scripts (-sC), and aggressive mode (-A adds OS detection and traceroute). This is your primary reconnaissance scan — document every open port, service, and version in your report.

  2. Perform banner grabbing with netcat on discovered ports
  3. Run dirb against the web services:
    dirb http://localhost
    What this does

    Concept: Brute-forces directory names on Brazos Financial’s web server. Look for admin panels, backup files, or configuration directories that shouldn’t be publicly accessible. Each finding is a potential vulnerability for your report.

  4. Execute nikto vulnerability scan:
    nikto -h http://localhost
    What this does

    Concept: Scans Brazos Financial’s web server for known misconfigurations, outdated software, and dangerous files. The findings from nikto complement your dirb results — nikto checks for specific vulnerabilities while dirb discovers hidden paths.

  5. Enumerate SMB with enum4linux:
    enum4linux -a localhost
    What this does

    Concept: The -a flag runs all enumeration checks: users, shares, groups, password policies, OS info. In Lab 1.2 you reviewed the help — now you’re running it for real against Brazos Financial’s SMB service on port 445. Document any discovered usernames, file shares, or policy weaknesses.

  6. Research vulnerabilities for discovered versions using searchsploit
  7. Document ALL findings with screenshots

Include in Part II (Client Deliverable)
  • Executive Summary (12%)
    • One page maximum. Hard limit — exceeding it is a Tier 2 deduction (−10%)
    • Overall risk posture in 1–2 sentences
    • Count of findings by severity (e.g., “3 Critical, 5 High, 8 Medium”)
    • Top 2–3 recommendations with business framing
    • Board-readable: no first person, no command output, no jargon unfamiliar to a non-technical executive

  • Methodology (7%)
    • 3–4 paragraph narrative of your engagement approach
    • Phases of the engagement (e.g., passive recon → active scanning → service enumeration → vulnerability identification)
    • Scope and limitations: what was in scope (authorized targets), what was not
    • Tools used at a high level (per-flag detail belongs in your Runbook, not here)
    • Written for the client, not for security peers — this is client-facing
  • Findings (22%)
    • Number each finding sequentially (Finding 1, Finding 2, ...)
    • Descriptive title (e.g., “SSH Service Exposes Outdated Software Version”)
    • Severity rating (Critical / High / Medium / Low / Informational)
    • Affected host(s) or service(s)
    • Description, including specific CVE or vulnerability identifiers where applicable
    • Evidence: reference to your screenshot(s) or tool output (e.g., “See Figure 4 in Appendix”)
    • Impact statement in business terms (what could happen to Brazos if this is exploited)
    • Per-finding recommendation summary (one line — full remediation lives in Recommendations)
    • Order findings by severity, highest first
  • Risk Analysis (11%)
    • Per-finding likelihood × impact analysis
    • Business context for Brazos specifically: financial services, regulatory exposure (SOC 2, PCI DSS where applicable), recent industry incidents
    • Answer “so what?” for each finding — why does this matter to Ray’s business
    • Differentiated reasoning per finding (avoid uniform “high risk” labels with no logic)
    • Reference an authoritative risk framework where helpful (NIST SP 800-30, OWASP risk rating, FAIR)
  • Recommendations (8%)
    • Prioritized: Monday-morning fixes (low effort, immediate impact) vs. next-quarter budget items (higher effort, strategic)
    • Each recommendation specific: what to do, what effort required, what priority window
    • Aligned with findings (each recommendation maps to one or more findings)
    • Acknowledge client constraints — “Ray has budget for 10 fixes, not 50; these are the top 10”
    • Where applicable, indicate which control framework the recommendation aligns with (CIS Controls, NIST CSF, etc.)

menu_book
Runbook Format: Like a Man Page

Per-tool entry, structured: Name (tool and basic purpose). Description (what it does, when you’d reach for it). Switches used (each flag you used on Brazos with a one-line explanation). Switches for future use (flags you didn’t use this time but should know). How you used it (your actual command against Brazos). What came back (the output, with screenshot reference). What you did with it (how the result fed into your findings). Bar: a colleague picking this up in 3 months should be able to reproduce your engagement.

psychology
Analyst Reflection: Honest, Not Performative

Real questions for real answers. Graders can tell when reflection is filler. Honest reflection earns credit — even short, honest reflection. Answer:

  • What did you take from this engagement specifically?
  • Did you have expectations going in? Verified or smashed?
  • Has this changed how you think about reconnaissance, consulting, or security?
  • Can you see how the work applies to the field outside this course?
  • What from this engagement will serve you in the future?
  • If you used AI tools: where, how, and how did you validate? If not, write N/A.
No minimum word count. No required structure. Genuine reflection.

What Red Team Learns
  • Open ports and service versions are your attack blueprint. Banner grabbing reveals exact software without triggering IDS. Directory discovery exposes admin panels and backup files that defenders forgot existed. SMB enumeration hands you a username list for credential attacks — before you’ve touched a single login form.

What Blue Team Learns
  • Your recon results are a defender’s punch list. Suppress version banners. Close ports that shouldn’t be public. Require authentication on SMB shares. Monitor for nmap, dirb, and nikto signatures in your logs — if you ran these tools undetected, so can an attacker.
