Deploy Brazos Container

What You’re Doing

Deploying the Brazos Financial Docker container that simulates their external attack surface. This is your authorized target for the engagement.

Why It Matters

Real penetration tests begin with scope definition and authorized target deployment. The Brazos container represents their actual exposed services — the same ports and configurations an external attacker would see.

  1. Pull the Brazos container image:
    docker pull jfnewsom/brazos-financial:latest
    What this does

    Concept: docker pull downloads a container image from Docker Hub. This image contains Brazos Financial’s simulated environment — a pre-configured system with intentional vulnerabilities for you to discover. The :latest tag ensures you get the current version.

  2. Run the container with all required ports:
    docker run -d -p 80:80 -p 443:443 -p 22:22 -p 445:445 -p 8443:8443 --name brazos jfnewsom/brazos-financial:latest
    What this does

    Concept: docker run creates and starts a container. The -d flag runs it in the background (detached). Each -p flag maps a container port to your host: 80 (HTTP), 443 (HTTPS), 22 (SSH), 445 (SMB), and 8443 (alternate HTTPS). These are the services you’ll scan — Brazos Financial’s exposed attack surface.

  3. Verify the container is running:
    docker ps
    What this does

    In your output: You should see a table with the brazos container listed as “Up.” Verify all five port mappings appear in the PORTS column. If the STATUS shows “Exited,” the container crashed — check the Common Pitfalls below.

  4. Note the container IP (usually localhost or 127.0.0.1)

Full Reconnaissance Workflow

What You’re Doing

Conducting comprehensive reconnaissance against the Brazos Financial container using all six tools from your training: nmap, netcat, dirb, nikto, enum4linux, and searchsploit.

Why It Matters

This is the core of your engagement. A thorough reconnaissance reveals everything an attacker would discover about Brazos Financial’s external attack surface. Your findings drive the entire client report.

  1. Run comprehensive nmap scan:
    nmap -sV -sC -A localhost
    What this does

    Concept: This combines all three Nmap techniques from Lab 1.2: version detection (-sV), default scripts (-sC), and aggressive mode (-A adds OS detection and traceroute). This is your primary reconnaissance scan — document every open port, service, and version in your report.

  2. Perform banner grabbing with netcat on discovered ports
  3. Run dirb against the web services:
    dirb http://localhost
    What this does

    Concept: Brute-forces directory names on Brazos Financial’s web server. Look for admin panels, backup files, or configuration directories that shouldn’t be publicly accessible. Each finding is a potential vulnerability for your report.

  4. Execute nikto vulnerability scan:
    nikto -h http://localhost
    What this does

    Concept: Scans Brazos Financial’s web server for known misconfigurations, outdated software, and dangerous files. The findings from nikto complement your dirb results — nikto checks for specific vulnerabilities while dirb discovers hidden paths.

  5. Enumerate SMB with enum4linux:
    enum4linux -a localhost
    What this does

    Concept: The -a flag runs all enumeration checks: users, shares, groups, password policies, OS info. In Lab 1.2 you reviewed the help — now you’re running it for real against Brazos Financial’s SMB service on port 445. Document any discovered usernames, file shares, or policy weaknesses.

  6. Research vulnerabilities for discovered versions using searchsploit
  7. Document ALL findings with screenshots
