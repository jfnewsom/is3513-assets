Section A: NEXUS Timesheet

Audience: NEXUS Management  •  Purpose: Billing Justification

Consulting firms bill clients for time, and every hour must be justified. Generic entries like “did lab work” won’t fly — describe what you actually did, which tools you used, and approximately how long each task took. Track time as you work, not after. Reconstructing hours days later leads to inaccurate entries that undermine your credibility.

Word: Professional Timesheet Tables

Create a table: Insert → Table → 4 columns (Week, Activity, Hours, Notes). Apply Table Design → Grid Table 4 - Accent 1 for professional alternating rows.

Auto-calculate totals: Click your Total cell → Table Layout → Formula → =SUM(ABOVE). Word calculates hours automatically and you never risk addition errors.

Repeating headers on multi-page tables: Select your header row → Table Layout → Repeat Header Rows. The column headers re-appear on every page without any extra work.

✓ Good Example: Nmap Service Detection — complete format with flags, output, next steps 
NMAP SERVICE VERSION DETECTION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Purpose:
Identify running services and their specific versions on target hosts.
Version info is critical for vulnerability research.

Basic Command:
  nmap -sV [target]

Recommended Flags:
  -sV                     Service version detection
  -sC                     Default scripts (safe recon)
  --version-intensity 5   Thorough probing (1–9 scale)
  -oN output.txt          Save to text file

Full Engagement Command:
  nmap -sV -sC --version-intensity 5 -oN brazos-svc.txt 10.0.0.0/24

Output Interpretation:
  PORT      STATE  SERVICE    VERSION
  22/tcp    open   ssh        OpenSSH 8.9p1 Ubuntu 3
  80/tcp    open   http       Apache httpd 2.4.29        ← flag this version
  3306/tcp  open   mysql      MySQL 5.7.42               ← EOL, research CVEs

Next Steps After Scan:
1. Record all service versions in findings documentation
2. Search NVD/CVE databases for known vulnerabilities per version
3. Note outdated versions for remediation recommendations
4. Cross-reference with client's patch management policy

Reference: Nmap Network Scanning, Gordon Lyon (nmap.org/book)
✓ Good Example: Hashcat Password Cracking Methodology — attack strategy with documentation guide 
HASHCAT PASSWORD CRACKING METHODOLOGY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Purpose:
Assess password strength by attempting to crack captured hashes.
Demonstrates real-world attack feasibility to the client.

Common Hash Modes:
  -m 0      MD5
  -m 1000   NTLM (Windows)
  -m 1800   sha512crypt (Linux /etc/shadow)
  -m 3200   bcrypt

Attack Strategy (run in order):
1. Dictionary attack — try common passwords first
   hashcat -m 1000 -a 0 hashes.txt rockyou.txt

2. Dictionary + Rules — add common mutations
   hashcat -m 1000 -a 0 hashes.txt rockyou.txt -r best64.rule

3. Mask attack — if password policy is known
   hashcat -m 1000 -a 3 hashes.txt ?u?l?l?l?l?d?d
   (Uppercase + 5 lowercase + 2 digits — e.g., Passwo12)

Documenting Results:
  • Total hashes provided: X
  • Cracked within 1 hour: Y (Z%)
  • Common patterns observed: [company name, seasons, years]
  • Policy gaps identified: [minimum length, complexity]

Ethical Note: Only crack hashes you have explicit authorization to test.
Document authorization in engagement scope.

Reference: Hashcat Wiki (hashcat.net/wiki) | NIST SP 800-63B

Word: Formatting Commands and Code Blocks

Monospace font for commands: Select your command text → change font to Consolas. This visually distinguishes commands from prose so readers can scan without reading every word.

Create a Code Block style: Format one code block perfectly → select the text → Home → Styles pane → Create a Style → name it "Code Block". Now apply it to every command with one click instead of reformatting each time.

Paste without formatting: When copying from terminal, use Ctrl+Shift+V (Paste Special → Unformatted Text) to avoid inheriting terminal font/color settings. Then apply your Code Block style.

Multi-level numbered lists for procedures: After typing a numbered item, press Tab to indent to a sub-step, Shift+Tab to go back up. If numbering restarts at 1 after a screenshot, right-click the number → Continue Numbering.

Section C: Client Report

Audience: Fictional Client Contact  •  Purpose: Professional Deliverable

Write the report you would hand to a real client. This is what Ray Jimenez, Jennifer Walsh, Amanda Nguyen, or Bobby Alamo would actually read. A professional security report has 13 components across three groups. Expand each section below for what to include and what to avoid.

Front Matter — Cover Page, Confidentiality & Disclaimer, Table of Contents

These three components set the professional tone before the client reads a single technical word.

1. Cover Page

Report title, client name, assessment dates, your firm name (NEXUS Security Consulting), and "CONFIDENTIAL" marking. Use Word's built-in cover page templates (Insert → Cover Page) rather than building from scratch. Set Different First Page in the header so page numbers don't appear on the cover.

2. Confidentiality & Disclaimer

Standard legal boilerplate — states the report is confidential, limits distribution to authorized personnel, notes that the assessment reflects conditions at a specific point in time, and caps liability. Once you write this once, save it as a Quick Part so you never type it again.

3. Table of Contents

Auto-generated from Heading styles — never typed manually. References → Table of Contents → Automatic Table 1. Requires that every section title uses a proper Heading 1/2/3 style. Update before submission with Ctrl+A then F9.

Executive Summary — one page, risk level first, severity table, 3 immediate actions

One page maximum. Written for executives who won't read technical details. A CEO should understand every word. Lead with the risk level in paragraph one — don't bury it.

Required elements:

  • Overall risk level stated immediately (HIGH / MODERATE / LOW)
  • Findings distribution table (Critical / High / Medium / Low counts)
  • 3 prioritized immediate actions — specific, not vague
SeverityCountWhat it means
Critical2Immediate exploitation risk — patch within 24–48 hours
High4Significant weakness — address within 2 weeks
Medium4Moderate risk — include in next patch cycle
Low2Minor hardening opportunities
Common mistakes:
  • Burying the overall risk assessment in paragraph three
  • Using technical jargon — "SSRF via X-Forwarded-For header" means nothing to a CFO
  • Vague action items like "improve security posture"
  • Running more than one page
Scope & Methodology — what was tested, what wasn't, and why the boundaries matter

Defines exactly what was (and was not) tested. Vague scope creates disputes. Clear scope protects everyone.

ElementWhat to include
In ScopeSpecific IP ranges, hostnames, applications, time window
Out of ScopeInternal networks, third-party services, DoS testing, social engineering
MethodologyReconnaissance → Enumeration → Vulnerability Analysis → Reporting
ConstraintsBusiness hours only, specific systems excluded at client request

"We tested the client's network and web applications" is not a scope statement. Which network? Which applications? No IP ranges = no verifiability = disputes later.

Risk Rating Scale — tie severity labels to CVSS scores so 'Critical' means something specific

Define what Critical, High, Medium, and Low mean in your report before using those words. Readers need an objective standard, not your personal opinion.

RatingCVSS ScoreExpected Action
Critical9.0–10.0Immediate exploitation possible — 24–48 hour remediation
High7.0–8.9Significant weakness — address within 2 weeks
Medium4.0–6.9Moderate risk — next patch cycle
Low0.1–3.9Best practice hardening — backlog item

Save this table as a Quick Part and drop it into every report. Adjust CVSS thresholds if the client's business context warrants it (e.g., healthcare clients may bump some Medium findings to High due to PHI exposure).

Attack Narrative — chain vulnerabilities into a story, not just a list of findings

The Attack Narrative shows how an attacker could chain your findings into a realistic attack path. It's where you demonstrate that you think like an attacker, not just a scanner operator.

Structure: Step 1 (Initial Access) → Step 2 (Privilege Escalation or Lateral Movement) → Step 3 (Data Access or Impact). Each step references a specific finding number.

End with a Business Impact statement that ties the full chain to consequences the client's leadership actually cares about — regulatory exposure, customer data, operational disruption.

What it is NOT:
  • A bullet list of your findings in order
  • "We found these vulnerabilities that could be exploited: Finding 1, Finding 2, Finding 3"
  • The same content as the Technical Findings section, repeated in prose
Technical Findings — the heart of the report, all six elements required per finding

The heart of the report. Every finding needs all six elements — missing one undermines credibility.

#ElementWhat to include
1Title & SeverityDescriptive name + Critical/High/Medium/Low badge + CVE if applicable
2Affected SystemsSpecific hosts, IPs, services — not "some servers"
3ObservationWhat you found, in third person past tense, no jargon
4EvidenceScreenshot with VM name visible, referenced as "Figure X"
5Business ImpactTie the finding to real consequences for THIS client's business
6RemediationSpecific, actionable, references a standard (CIS, NIST)

Client context matters. The same vulnerability gets written differently depending on who's reading:

  • Brazos Financial — frame risk in terms of SEC cybersecurity rule compliance and AUM exposure
  • Gulf Coast Healthcare — connect findings to HIPAA Security Rule sections and OCR audit risk
  • LoneStar DevOps — tie to Series A/enterprise customer trust and CI/CD pipeline security
  • Alamo Industries — reference CMMC requirements and IT/OT convergence risks
Classic incomplete finding:
  • Title: "Old Software" — no severity, no CVE, no affected systems
  • Body: "The server is running old software that should be updated." — no version info, no evidence, no impact, no specific remediation
  • The client can't act on this. It wastes everyone's time.
Security Strengths — what the client is doing right, with specific evidence

Not fluff — it reinforces good practices and gives a balanced report. A report with nothing positive feels biased.

Strengths need evidence, just like findings. "The company appears to take security seriously" is not a strength observation — it's a placeholder.

  • Good: "MFA is enforced on all VPN connections and administrative interfaces" — specific, observable, verifiable
  • Good: "Patch compliance for external systems is at 94% based on the January 3 Nessus scan" — cited, measurable
  • Weak: "They have some good practices in place" — tells the client nothing about what to keep doing
Recommendations — tiered by timeline (Immediate / Short-term / Long-term) with standards references

Actionable, prioritized, realistic. A client who paid for an assessment deserves specific guidance — not "update your software."

Tiered structure:

TierTimelineExample
Immediate0–30 daysPatch Apache to 2.4.58; disable TLS 1.0/1.1 on public services; rotate exposed credentials
Short-term30–90 daysDeploy WAF; implement automated patch management; enforce password complexity policy
Long-term90+ daysAdopt secrets management (HashiCorp Vault); establish annual pentest cadence; develop IR playbooks

Always reference an industry standard (CIS Controls, NIST CSF, NIST SP 800-53) to back up recommendations. A recommendation with no framework citation is just an opinion.

Screenshots & Evidence — VM name in every screenshot, auto-numbered captions, cross-referenced in text

Every claim needs proof. Screenshots without VM names are unverifiable — see the Screenshot Requirements page for the full standard. Key rules here:

  • VM name (kali-lastname-studentID) must be visible in the terminal prompt in every screenshot
  • File naming: ##-description.png (e.g., 07-nmap-service-scan.png)
  • Embed screenshots inline — never link to external files
  • Every figure gets a caption (Word auto-numbering) and is referenced in the text (Word cross-reference)
  • Annotate screenshots to highlight the key finding — draw a box, add an arrow, bold the relevant output line

Screenshots go in the Findings body. Raw terminal output goes in Appendix A. Don't mix them.

Appendices & References — organize raw output, cite 10+ sources, never type a bibliography manually

Appendix A: Raw Tool Output

Paste raw text output here — full nmap scans, hashcat session logs, Wireshark statistics. Label each block clearly ("Appendix A1: Full Nmap Version Scan, January 15 14:32 CST"). Reference from the body: "See Appendix A1." Screenshots do NOT go here.

References (Minimum 10)

See the Citations page for the full guide. Quick reminder of what counts:

  • NIST publications, CIS Benchmarks, OWASP guides, CVE entries
  • Tool documentation (nmap.org/book, hashcat.net/wiki, Wireshark docs)
  • Course textbook (Conklin & White) — cite the specific chapter
  • Man pages, RFC documents, vendor security advisories
  • Google is a search engine, not a source. Wikipedia is a starting point, not a citation.

Word: Finding Templates and Consistent Formatting

Save a finding template as a Quick Part: Format one finding perfectly (title, severity badge, all six subsections). Select it all → Insert → Quick Parts → Save Selection → name it "Finding Template". Clients notice when Finding #1 looks different from Finding #7.

Severity badges with text boxes: Insert → Text Box → Simple Text Box. Set fill color (red for Critical), white text, no outline. Right-click → Wrap Text → In Front of Text to position it in the finding header.

Page breaks between findings: Place cursor before each new finding → Ctrl+Enter. Each finding on its own page reads more professionally than findings running together.

SmartArt for attack chains: Insert → SmartArt → Process → Basic Chevron Process. Each chevron = one step in the attack path. Color-code by severity. Group shapes with Ctrl+G so they move as one unit.

Word Automation Quick Reference

Master these features and your reports will look professional with a fraction of the formatting effort. Every one of these is a skill you'll use in professional practice — not just for this course. Expand what you need, when you need it.

Document Setup — auto-populate your name in headers, footers, and cover page

Set this once and your name auto-populates the cover page, headers, and footers in every report you create from the IS3513 template.

Windows: Set User Information

  1. File → Options → General
  2. Under Personalize your copy of Microsoft Office — enter your User name and Initials
  3. Click OK, then close and reopen your document

Mac: Set User Information

  1. Word → Preferences → User Information
  2. Enter your Name and Initials
  3. Close Preferences, then close and reopen your document

Verify your fields are working

Right-click your name on the cover page → Toggle Field Codes. You should see { AUTHOR } or { TITLE } rather than plain typed text. Toggle again to return to normal view.

If the cover still shows a placeholder after setting user info: Ctrl+A then F9 (Windows) or Cmd+A then fn+F9 (Mac). Save, close, reopen.

Headings → Automatic Table of Contents — never type a TOC manually

Never type a Table of Contents manually. Word builds it from your Heading styles and updates it in seconds. A hand-typed TOC breaks every time you add content.

Step 1 — Apply Heading Styles

Select each section title → Home → Styles → Heading 1 for main sections, Heading 2 for subsections. Shortcut: Ctrl+Alt+1 / Ctrl+Alt+2. Never just make text big and bold — use actual styles or the TOC won't see it.

Step 2 — Insert the TOC

Place cursor on the TOC page → References → Table of Contents → Automatic Table 1. Word builds it instantly.

Step 3 — Update before submitting

Ctrl+A → F9 → choose Update entire table. This refreshes page numbers whenever you've added or rearranged content.

Bonus: TOC entries are clickable hyperlinks — Ctrl+Click to jump to any section. Your PDF reader will have the same navigation.

Figure Captions & Cross-References — auto-numbered figures that update when you rearrange

Never type "Figure 1" manually. Word tracks figure numbers automatically. Add an image, delete an image, reorder — all numbers update with one keystroke.

Adding a caption

  1. Click your screenshot or image
  2. References → Insert Caption
  3. Label: Figure, then type a description — e.g., "Nmap service scan results showing Apache 2.4.29"
  4. Word adds "Figure 1:" and auto-increments for every subsequent image

Referencing a figure in text

  1. Place cursor where "Figure X" should appear
  2. References → Cross-reference
  3. Reference type: Figure → select your figure → check Insert as hyperlink → Insert

Update all at once: Ctrl+A → F9 — this is the same update-all command as the TOC. Do it once before every submission and everything refreshes.

"Error! Reference source not found" = you typed the figure number manually. Delete it and use Cross-reference instead.

Citation Manager & Bibliography — 10 sources, auto-formatted, zero manual typing

Never type a bibliography manually. Word's citation manager formats every source automatically and generates the reference list with one click.

Adding a source

  1. References → Manage Sources → New
  2. Choose Type of Source (Website, Book, Report, etc.)
  3. Fill in Author, Title, Year, URL — sources save for reuse across documents

Inserting an in-text citation

Place cursor after your claim → References → Insert Citation → select your source. Word adds "(Author, Year)" automatically. Build your source library as you write — don't wait until the end.

Generating the bibliography

Cursor at end of document → References → Bibliography → Insert Bibliography. Done. Update with Ctrl+A → F9 before submitting.

What counts as a citable source:

  • NIST publications, CIS Benchmarks, OWASP guides
  • CVE entries from nvd.nist.gov
  • Tool documentation (Nmap book, Hashcat wiki, Wireshark docs)
  • The course textbook (Conklin & White)
  • Man pages, vendor docs, RFC documents

Google is a search engine, not a source. Wikipedia is a starting point, not a citation. Minimum 10 sources — aim for quality over quantity.

Tables — professional timesheet tables, =SUM(ABOVE), color-coded severity cells

Creating a professional table

  1. Insert → Table → choose grid size
  2. Click inside table → Table Design tab appears → choose a style ("Grid Table 4 - Accent 1" gives professional alternating rows)
  3. Check Header Row and Banded Rows

Keep header row visible on multi-page tables

Select the header row → Table Layout → Repeat Header Rows. The header re-appears automatically on every page.

Auto-calculate timesheet totals

Click in the "Total" cell → Table Layout → Formula → enter =SUM(ABOVE). Word calculates your hours automatically. Update with F9 if you change values.

Color-code severity in findings tables

Click in severity cell → Table Design → Shading → Red (#C00000) for Critical, Orange (#E26B0A) for High, Yellow (#FFFF00) for Medium, Green (#70AD47) for Low. White text on red/orange; black on yellow/green.

Quick Parts — save your disclaimer, finding template, scope table; reuse across reports

Quick Parts let you save any formatted block — your disclaimer, a finding template, your scope table — and insert it anywhere with two clicks. Build your library once and reuse it across every engagement.

Saving a block

  1. Select the formatted content you want to reuse (disclaimer, finding template, etc.)
  2. Insert → Quick Parts → Save Selection to Quick Part Gallery
  3. Give it a descriptive name (e.g., "Security Report Disclaimer")

Inserting a saved block

Insert → Quick Parts → click the saved item. It drops in with full formatting intact.

Good candidates for Quick Parts: confidentiality disclaimer, finding template (with all 6 subsections), scope table skeleton, risk rating table, remediation timeline table, NEXUS Security boilerplate intro.

Keyboard Shortcuts & Troubleshooting — the 6 you'll use every report + common fixes
Ctrl+A, F9

Update ALL fields — TOC, figure numbers, cross-references, bibliography. Run this before every submission.

Ctrl+Enter

Page break. Never hit Enter 20 times to start a new page — this breaks whenever you edit content above.

Ctrl+Shift+V

Paste without formatting. Use when copying from terminal or the web to prevent font/spacing disasters.

Ctrl+Alt+1/2/3

Apply Heading 1/2/3 instantly. Required for auto-TOC to work. Faster than clicking the Styles pane.

Ctrl+D

Font dialog — change font, size, and color in one place. Good for applying Consolas to command blocks.

Cmd+A, fn+F9

Mac equivalent of Ctrl+A, F9. Update all fields on macOS. fn key required on most Mac keyboards.

Troubleshooting quick hits:

  • TOC shows wrong page numbers: Ctrl+A → F9 → Update entire table
  • Numbered list restarts at 1: Right-click the number → Continue Numbering
  • Table splits awkwardly across pages: Select header row → Table Layout → Repeat Header Rows
  • Figure caption separated from image: Select image → Home → Paragraph → Keep with next
  • Can't delete blank page at end: Home → Show/Hide ¶ → delete the paragraph marks you see there
