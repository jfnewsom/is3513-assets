Section A: NEXUS Timesheet
Audience: NEXUS Management | Purpose: Billing Justification

What You're Doing

Documenting all hours across the three-week engagement. Consulting firms bill clients for time, and analysts must justify every hour. This is standard practice.

What to Include

Week Billing Type Content
Week 1 Internal Training Environment setup, initial tool learning
Week 2 Internal Training Tool exploration, technique practice
Week 3 Client Billable Engagement execution, report writing
Example Timesheet Entry 
Week 1 - Internal Training (3.5 hours)
- Kali VM configuration and validation: 1.0 hr
- Network scanning tool installation: 0.5 hr  
- Initial reconnaissance methodology review: 1.0 hr
- Documentation and screenshot capture: 1.0 hr

Week 2 - Internal Training (4.0 hours)
- Nmap advanced scanning techniques: 1.5 hr
- Banner grabbing and service enumeration: 1.0 hr
- OSINT tool exploration: 1.0 hr
- Runbook documentation: 0.5 hr

Week 3 - Client Billable (5.0 hours)
- Brazos Financial external reconnaissance: 2.0 hr
- Finding analysis and risk assessment: 1.5 hr
- Client report drafting: 1.5 hr

TOTAL: 12.5 hours (7.5 internal + 5.0 billable)
Common Pitfalls: Generic descriptions like "did lab work" | Unrealistic times (2 hours for a full scan) | Missing internal training weeks
Section B: Internal Runbook
Audience: NEXUS Team | Purpose: Knowledge Transfer

What You're Doing

Creating methodology documentation so another analyst could reproduce your work. "Here's how to do X for the next analyst." When you leave, your methodology stays.

What to Include

From Week 1
  • Environment setup steps
  • Tool installation procedures
  • Configuration documentation
  • Screenshots of key steps
From Week 2
  • Tool usage documentation
  • Command syntax with explanations
  • Technique workflows
  • Output interpretation guides
From Week 3
  • Engagement workflow
  • Lessons learned
  • What worked/didn't work
  • Recommendations for next time

The Reproducibility Test

Ask yourself: "Could a junior analyst who wasn't in this class follow my runbook and achieve the same results?" If the answer is no, add more detail.

Example Runbook Entry (Tool Documentation) 
Nmap Service Version Detection

Purpose: Identify running services and their versions on target hosts.

Basic Command:
nmap -sV [target]

Recommended Flags:
-sV          Service version detection
-sC          Default scripts (safe)
--version-intensity 5   Thorough version probing
-oN output.txt         Save to file

Full Command Used:
nmap -sV -sC --version-intensity 5 -oN brazos-scan.txt 10.0.0.0/24

Output Interpretation:
- PORT column: Open ports discovered
- STATE: open/closed/filtered
- SERVICE: Identified service name
- VERSION: Specific version (critical for vulnerability research)

Screenshot: See 07-nmap-service-scan.png

Reference: Nmap Network Scanning, Gordon Lyon (nmap.org/book)
Pro Tip: If you followed directions in Labs X.1 and X.2, your runbook is 60-70% complete before Week 3 starts. Just compile and polish.
Section C: Client Report
Audience: Fictional Client Contact | Purpose: Professional Deliverable

What You're Doing

Writing a professional security report that you would hand to a real client. This is what the client contact (Ray Jimenez, Jennifer Walsh, etc.) would read.

Required Sections

1. Executive Summary (1 page max)
High-level overview for non-technical readers. What did you find? What's the risk? What should they do? No jargon. A CEO should understand this.
2. Scope and Methodology
What systems were tested? What was out of scope? What approach did you take? This sets boundaries and expectations.
3. Findings
What did you discover? Each finding should include: description, evidence (screenshots), severity, and business impact. This is where your screenshots live.
4. Risk Analysis
What's the actual risk to the business? Consider likelihood, impact, and business context. Not everything high-severity is high-priority.
5. Recommendations
Actionable remediation steps. Prioritized. Realistic. Include quick wins and long-term improvements. Reference industry standards (CIS, NIST).

Writing Quality Expectations

Do This
  • Write in third person, past tense
  • Use clear, professional language
  • Explain technical terms
  • Support claims with evidence
  • Proofread for grammar/spelling
Avoid This
  • First person ("I ran nmap...")
  • Unexplained jargon
  • Claims without screenshots
  • Copy-pasted tool output without analysis
  • Casual language or slang
Example Finding Entry 
Finding 3: Outdated Apache Web Server

Severity: High
CVSS Score: 7.5

Description:
The external web server at 203.0.113.10 is running Apache HTTP Server 
version 2.4.29. This version was released in October 2017 and contains 
multiple known vulnerabilities including CVE-2021-44790 (buffer overflow) 
and CVE-2021-41773 (path traversal).

Evidence:
Service version was identified through banner grabbing and confirmed 
via Nmap service detection. See Figure 7 for scan results.

[Screenshot: 12-apache-version-evidence.png]

Business Impact:
An attacker could exploit these vulnerabilities to execute arbitrary code 
on the web server, potentially gaining access to customer financial data 
and internal network resources.

Recommendation:
Upgrade Apache HTTP Server to the latest stable version (2.4.58 as of 
this assessment). Implement a patch management policy to ensure timely 
updates. Reference: CIS Apache HTTP Server Benchmark v2.4.
