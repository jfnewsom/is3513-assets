Word Automation: Professional Tables for Timesheets

Create clean timesheet tables: Insert → Table → Choose 4 columns (Week, Activity, Hours, Notes). Word's table tools handle alignment automatically.

Apply a professional style: Click anywhere in your table → Table Design tab appears → Choose a style from the gallery. "Grid Table 4 - Accent 1" gives you professional alternating rows.

Auto-calculate totals: In the final "Total" cell, go to Layout → Formula → Enter =SUM(ABOVE). Word calculates your hours automatically.

Keep header row visible: Select your header row → Layout → check Repeat Header Rows. If your table spans pages, the header appears on each page.

Mastering Word tables saves 5-10 minutes per timesheet and ensures consistent, professional formatting across all your submissions.

What You're Doing

Creating methodology documentation so another analyst could reproduce your work. "Here's how to do X for the next analyst." When you leave the firm, your methodology stays behind.

It's Really for Future You

Here's the truth: you will forget what you did here. Three months from now, you'll face a similar problem and think "I know I figured this out before..." but you won't remember the exact commands, the specific flags, or the workarounds you discovered.

Finding the right solution often means Googling multiple sources. A little bit from this StackOverflow answer, a little bit from that blog post, some trial and error, and finally — YOUR solution that actually works in your environment.

Document it so you don't have to reinvent the wheel next time. Your runbook isn't just for "the next analyst" — it's for future you, who will thank present you for writing it down.

The Reproducibility Test

Ask yourself: "Could a junior analyst who wasn't in this class follow my runbook and achieve the same results?" If the answer is no, add more detail.

NMAP SERVICE VERSION DETECTION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Purpose:
Identify running services and their specific versions on target hosts.
Version information is critical for vulnerability research.

Basic Command:
┌────────────────────────────────────────────────────────────┐
│  nmap -sV [target]                                         │
└────────────────────────────────────────────────────────────┘

Recommended Flags:
  -sV                     Service version detection
  -sC                     Default scripts (safe reconnaissance)
  --version-intensity 5   Thorough version probing (1-9 scale)
  -oN output.txt          Save results to text file
  -oX output.xml          Save results to XML (for parsing)

Full Engagement Command:
┌────────────────────────────────────────────────────────────┐
│  nmap -sV -sC --version-intensity 5 -oN brazos-svc.txt \   │
│       -oX brazos-svc.xml 10.0.0.0/24                       │
└────────────────────────────────────────────────────────────┘

Output Interpretation:
  PORT      STATE  SERVICE    VERSION
  ────      ─────  ───────    ───────
  22/tcp    open   ssh        OpenSSH 8.9p1 Ubuntu 3
  80/tcp    open   http       Apache httpd 2.4.29
  443/tcp   open   ssl/http   Apache httpd 2.4.29
  3306/tcp  open   mysql      MySQL 5.7.42

  • PORT: Open ports discovered
  • STATE: open/closed/filtered
  • SERVICE: Identified service name
  • VERSION: Specific version (use this for CVE research)

Next Steps After Scan:
1. Record all service versions in findings documentation
2. Search CVE databases for known vulnerabilities
3. Note outdated versions for remediation recommendations
4. Cross-reference with client's patch management policy

✓ Good Example: OpenSSL Certificate Analysis Runbook Entry 
OPENSSL CERTIFICATE ANALYSIS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Purpose:
Retrieve and analyze SSL/TLS certificates from remote servers to identify 
expiration dates, weak algorithms, and trust chain issues.

Basic Certificate Retrieval:
┌───────────────────────────────────────────────────────────────┐
│  openssl s_client -connect [host]:443 -showcerts              │
└───────────────────────────────────────────────────────────────┘

View Certificate Details:
┌───────────────────────────────────────────────────────────────┐
│  echo | openssl s_client -connect [host]:443 2>/dev/null \ │
│       | openssl x509 -noout -text                             │
└───────────────────────────────────────────────────────────────┘

Check Expiration Date Only:
┌───────────────────────────────────────────────────────────────┐
│  echo | openssl s_client -connect [host]:443 2>/dev/null \ │
│       | openssl x509 -noout -dates                            │
└───────────────────────────────────────────────────────────────┘

Key Fields to Document:
  • Issuer:           Who issued the certificate (CA)
  • Subject:          What domain/entity it's issued to
  • Not Before:       Valid from date
  • Not After:        Expiration date ← CRITICAL
  • Public Key:       Algorithm and key size (RSA 2048+ required)
  • Signature Alg:    Should be SHA-256 or stronger (not SHA-1)

Red Flags to Report:
  ✗ Expired certificates
  ✗ Self-signed certificates on public-facing services
  ✗ SHA-1 signatures (deprecated)
  ✗ RSA keys smaller than 2048 bits
  ✗ Wildcard certificates on internal systems
  ✗ Mismatched Common Name (CN doesn't match hostname)

✓ Good Example: Password Cracking Methodology Runbook Entry 
HASHCAT PASSWORD CRACKING METHODOLOGY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Purpose:
Assess password strength by attempting to crack captured hashes.
Demonstrates real-world attack feasibility.

Hash Identification:
┌────────────────────────────────────────────────────────────┐
│  hashid [hash_value]                                       │
│  # or check hash length and format manually                │
└────────────────────────────────────────────────────────────┘

Common Hash Modes:
  -m 0      MD5
  -m 100    SHA1
  -m 1000   NTLM (Windows)
  -m 1800   sha512crypt (Linux /etc/shadow)
  -m 3200   bcrypt

Attack Strategy (in order):

1. Dictionary Attack - Try common passwords first
┌────────────────────────────────────────────────────────────┐
│  hashcat -m 1000 -a 0 hashes.txt rockyou.txt               │
└────────────────────────────────────────────────────────────┘

2. Dictionary + Rules - Add common mutations
┌────────────────────────────────────────────────────────────┐
│  hashcat -m 1000 -a 0 hashes.txt rockyou.txt -r best64.rule│
└────────────────────────────────────────────────────────────┘

3. Mask Attack - If password policy is known
┌────────────────────────────────────────────────────────────┐
│  hashcat -m 1000 -a 3 hashes.txt ?u?l?l?l?l?l?d?d          │
│  # Uppercase + 5 lowercase + 2 digits (e.g., Passwo12)     │
└────────────────────────────────────────────────────────────┘

Mask Characters:
  ?l  lowercase    ?u  uppercase    ?d  digit
  ?s  special      ?a  all chars    ?b  byte (0x00-0xff)

Documenting Results:
  • Total hashes provided: X
  • Cracked within 1 hour: Y (Z%)
  • Common patterns observed: [list]
  • Weakest passwords found: [list - sanitized]

Ethical Note:
Only crack hashes you have explicit authorization to test.
Document authorization in engagement scope.

✓ Good Example: Wireshark Packet Analysis Runbook Entry 
WIRESHARK PACKET ANALYSIS METHODOLOGY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Purpose:
Analyze network traffic captures (PCAP files) to identify anomalies,
protocol issues, and potential indicators of compromise.

Opening a Capture File:
┌────────────────────────────────────────────────────────────┐
│  wireshark -r capture.pcap                                 │
│  # or File → Open within GUI                          │
└────────────────────────────────────────────────────────────┘

Essential Display Filters:

  By Protocol:
  ┌──────────────────────────────────────────────────────────┐
  │  http             HTTP traffic only                      │
  │  dns              DNS queries and responses              │
  │  tcp              All TCP traffic                        │
  │  tcp.flags.syn==1 SYN packets (connection attempts)      │
  └──────────────────────────────────────────────────────────┘

  By Address:
  ┌──────────────────────────────────────────────────────────┐
  │  ip.addr == 10.0.0.50        Traffic to/from host        │
  │  ip.src == 10.0.0.50         Traffic FROM host           │
  │  ip.dst == 10.0.0.50         Traffic TO host             │
  └──────────────────────────────────────────────────────────┘

  By Port:
  ┌──────────────────────────────────────────────────────────┐
  │  tcp.port == 443             HTTPS traffic               │
  │  tcp.port == 4444            Common Metasploit port      │
  │  udp.port == 53              DNS traffic                 │
  └──────────────────────────────────────────────────────────┘

  Combined Filters:
  ┌──────────────────────────────────────────────────────────┐
  │  http && ip.src == 10.0.0.50                     │
  │  # HTTP traffic from specific source                     │
  └──────────────────────────────────────────────────────────┘

Key Analysis Steps:
1. Statistics → Protocol Hierarchy (see what's in the capture)
2. Statistics → Conversations (identify talkers)
3. Statistics → Endpoints (list all hosts)
4. Filter for anomalies (unusual ports, high volume)
5. Follow TCP Stream for suspicious connections

Following a TCP Stream:
  Right-click packet → Follow → TCP Stream
  Shows full conversation in readable format

Red Flags to Document:
  ✗ Connections to known bad IPs (check threat intel)
  ✗ Unusual ports (4444, 5555, non-standard HTTP)
  ✗ Large data transfers to external hosts
  ✗ Repeated failed connection attempts (scanning)
  ✗ DNS queries to suspicious domains
  ✗ Unencrypted credentials in cleartext protocols

✓ Good Example: tcpdump Command-Line Capture Runbook Entry 
TCPDUMP COMMAND-LINE CAPTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Purpose:
Capture network traffic from command line when GUI isn't available.
Essential for remote server analysis and scripted captures.

Basic Capture (requires root):
┌────────────────────────────────────────────────────────────┐
│  sudo tcpdump -i eth0                                      │
│  # Capture on interface eth0, output to terminal           │
└────────────────────────────────────────────────────────────┘

Save to File for Wireshark Analysis:
┌────────────────────────────────────────────────────────────┐
│  sudo tcpdump -i eth0 -w capture.pcap                      │
│  # -w writes to file instead of terminal                   │
└────────────────────────────────────────────────────────────┘

Essential Flags:
  -i [interface]    Specify interface (eth0, wlan0, any)
  -w [file]         Write to PCAP file
  -r [file]         Read from PCAP file
  -c [count]        Stop after N packets
  -n                Don't resolve hostnames (faster)
  -nn               Don't resolve hostnames or ports
  -v, -vv, -vvv     Increasing verbosity

Filtering Captures:

  By Host:
  ┌──────────────────────────────────────────────────────────┐
  │  sudo tcpdump -i eth0 host 10.0.0.50                     │
  │  sudo tcpdump -i eth0 src 10.0.0.50                      │
  │  sudo tcpdump -i eth0 dst 10.0.0.50                      │
  └──────────────────────────────────────────────────────────┘

  By Port:
  ┌──────────────────────────────────────────────────────────┐
  │  sudo tcpdump -i eth0 port 80                            │
  │  sudo tcpdump -i eth0 port 443 or port 80                │
  └──────────────────────────────────────────────────────────┘

  By Protocol:
  ┌──────────────────────────────────────────────────────────┐
  │  sudo tcpdump -i eth0 tcp                                │
  │  sudo tcpdump -i eth0 udp                                │
  │  sudo tcpdump -i eth0 icmp                               │
  └──────────────────────────────────────────────────────────┘

Production Capture Command:
┌────────────────────────────────────────────────────────────┐
│  sudo tcpdump -i eth0 -nn -w engagement.pcap \             │
│       -c 100000 'not port 22'                              │
│  # Capture 100k packets, exclude SSH, no DNS resolution    │
└────────────────────────────────────────────────────────────┘

Workflow:
1. Identify correct interface: ip addr or ifconfig
2. Start capture with appropriate filters
3. Let run during test period
4. Stop with Ctrl+C
5. Transfer PCAP to analysis workstation
6. Open in Wireshark for detailed analysis

Reference: 
tcpdump man page (tcpdump.org)
SANS tcpdump Cheat Sheet

Troubleshooting Common Problems

❌ Problem: TOC shows wrong page numbers

Fix: Select all (Ctrl+A) then update fields (F9). Also right-click TOC → "Update Table" → "Update entire table".

❌ Problem: Numbered list restarts at 1

Fix: Right-click the number → "Continue Numbering" or "Set Numbering Value" to manually set the starting number.

❌ Problem: Weird fonts after paste from terminal

Fix: Use Ctrl+Shift+V to paste without formatting, then apply your code style.

❌ Problem: Table splits awkwardly across pages

Fix: Select header row → Layout → Repeat Header Rows. For rows that shouldn't split: Select row → Paragraph → Keep with next.

❌ Problem: Can't delete extra blank page at end

Fix: Go to the blank page → Home → Show/Hide ¶ → Delete the paragraph marks (¶) you see there.

❌ Problem: Figure caption and image separated

Fix: Select the image → Home → Paragraph dialog → Keep with next. This prevents page breaks between image and caption.
